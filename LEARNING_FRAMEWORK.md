## ① Why - 价值 (为什么)

**目标**：解决什么问题？带来什么价值？

### 必答问题
- 背景是什么？不用它有什么痛点？
- 用了它能带来什么收益？（效率↑/成本↓/体验↑）
- 谁会用到这个功能？

### 大白话案例
```
比如：原来每天手动同步数据到凌晨 3 点，用了定时任务后，
系统自动跑，第二天上班数据已经同步好了，运营可以直接看报表。
```

## ③ How - 思维 (怎么做)

**目标**：如果我们自己做，怎么实施？数据模型？

### 必答问题

**目标**：我想要实现/修复什么？
```
例：实现一个分布式定时任务调度功能
```

**范围**：只允许动哪些目录/文件？
```
例：
- yudao-module-job/                      (定时任务模块)
- yudao-framework/yudao-spring-boot-starter-job/  (框架封装)
禁止动：
- yudao-module-system/                   (系统模块)
- yudao-framework/yudao-spring-boot-starter-security/ (安全模块)
```

**禁止**：哪些目录/文件/行为绝对不要碰？
```
例：
- 禁止修改数据库表结构
- 禁止引入新的外部依赖
- 禁止改动已有 API 接口
```

### 数据模型设计

**核心问题**：要设计几张表？每张表有什么字段？

**数据模型示例**：
```
定时任务表 (xxl_job)：
- job_id        : 任务ID
- name         : 任务名称
- cron_expression : Cron表达式
- handler_name : 执行器方法名
- status       : 状态(0停用/1启用)
- tenant_id    : 租户ID
- created_time : 创建时间
- updated_time: 更新时间

执行日志表 (xxl_job_log)：
- log_id       : 日志ID
- job_id       : 任务ID
- execute_time : 执行时间
- duration    : 执行耗时
- status      : 状态(成功/失败)
- error_msg   : 错误信息
```

### 关键流程设计

**核心流程**：从入口到结果的完整流程

```
流程示例：
调度中心(cron触发) → 检查任务状态 → 选一台执行器 → 下发执行指令 → 
执行器接收 → 执行业务逻辑 → 记录执行日志 → 返回结果
```

### 关键代码设计

**核心问题**：关键方法/类有哪些？

```
关键类：
- XxlJobScheduler    : 调度器核心类
- XxlJobExecutor    : 执行器核心类  
- XxlJobService     : 任务管理服务
- XxlJobRepository  : 数据访问层
```

## ④ Hard - 难点 (挑战)

**目标**：挑战和坑在哪？举例子

### 必答问题
- 可能遇到什么问题？
- 边界情况有哪些？
- 有什么坑？

### 大白话案例

**难点1：多台机器同时跑，任务重复执行？**
```
场景：A服务部署了3台，每台都有定时任务代码。
到了9点，3台同时触发，都去执行任务，导致执行3次。
→ 需要分布式锁抢锁，只有一台能抢到并执行
```

**难点2：任务执行一半挂了？**
```
场景：任务要同步10000条数据，同步到5000条时服务器重启了。
→ 需要补偿机制，记录进度，自动重试从5001条开始
```

**难点3：想改任务参数不想重启？**
```
场景：运营想把同步时间从9点改成10点。
→ 需要热更新，从数据库读取配置，修改立即生效
```

**难点4：任务执行超时？**
```
场景：任务应该在1分钟内完成，但业务逻辑太复杂跑了1小时。
→ 需要超时告警，自动中断任务
```

## ⑤ Metric - 衡量 (指标)

**目标**：衡量这个功能做得好不好的指标

### 指标设计原则
- 不是固定指标，是基于**这个业务功能**设计的指标
- 每个指标都要有明确的**权重**
- 每个指标都要有**验证方法**

### 指标设计模板

| 指标 | 权重 | 说明 | 怎么验证 |
|------|------|------|----------|
| 功能完整性 | 20% | 需求覆盖程度 | 逐条核对需求文档 |
| 可用性 | 20% | 系统稳定运行 | 24h运行无报错 |
| 性能 | 15% | 执行时间/吞吐 | 压测报告 |
| 可维护性 | 15% | 代码清晰度 | Code Review |
| 扩展性 | 15% | 能否灵活扩展 | 新需求开发时间 |
| 安全性 | 15% | 权限/数据安全 | 安全测试报告 |

### 大白话案例

**定时任务衡量指标**（业务定制）：
```
| 指标                    | 权重 | 说明                       | 验证方法        |
|------------------------|------|----------------------------|----------------|
| 任务执行成功率≥99.9%    | 30%  | 成功执行次数/总次数         | 日志统计       |
| 任务重复执行次数=0       | 25%  | 同一时间只执行一次         | 日志+分布式锁 |
| 任务修改生效时间≤1min    | 20%  | 修改配置到生效时间          | 手动测试      |
| 任务执行日志可查          | 15%  | 日志保留30天              | 后台查询      |
| 支持动态修改不重启       | 10%  | 改配置不用重启服务         | 手动测试      |
```

**目标**：业界有哪些方案？怎么选？

- 候选方案有哪些？(至少3个)

### 选型对比表

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| XXL-Job | 文档全、管理后台、支持分布式 | 需额外部署 | 中大型项目 |
| ElasticJob | 支持分片、代码简洁 | 社区较小 | 需要分片场景 |

| Quartz | 通用、久经考验 | 配置复杂 | 传统项目 |

### 选型理由模板
```
最后选 XXL-Job，因为：
1. 文档最全，社区活跃，遇到问题能搜到答案
2. 有管理后台，运营可以直接配置任务
3. 支持分布式锁，解决重复执行问题
4. 与项目技术栈匹配(Spring Boot)
```

## ⑦ Impl - 实现 (细节)

**目标**：项目内具体实现是什么？

### 必答问题

#### 数据模型

**核心实体类**：
```java
public class XxlJob {
    private Long id;
    private String name;
    private String cron;
    private String handler;
    private Integer status;
    private Long tenantId;
    private Date createdTime;
    private Date updatedTime;
}
```

#### 核心代码

**执行器代码**：
```java
@XxlJob("demoJob")
public void execute() {
    // 1. 执行业务逻辑
    // 2. 记录日志
    // 3. 返回结果
}
```

#### 关键步骤校验

```
Step 1: 检查任务状态
  → 校验点：status == 1
  → 验证方法：查询数据库

Step 2: 抢分布式锁
  → 校验点：SETNX返回OK
  → 验证方法：Redis返回值

Step 3: 执行任务
  → 校验点：返回成功
  → 验证方法：业务执行结果
```

#### 失败恢复机制

```
场景1：任务执行失败
  → 触发：重试机制(最多重试3次)
  → 恢复：记录错误日志，发送告警

场景2：服务重启
  → 触发：补偿机制
  → 恢复：读取断点日志，从断点继续

场景3：分布式锁失效
  → 触发：锁超时释放
  → 恢复：其他节点抢锁执行
```

#### 约束条件

```
- 不引入新依赖？（是，使用已有依赖）
- 保持 API 兼容？（是，不改变接口签名）
- 性能要求：单次执行<1分钟
- 安全要求：任务参数需脱敏
```

## ⑧ SKILL - 提炼 (复用)

**目标**：将能力提炼为可复用 SKILL

### 触发条件

什么场景下用这个 SKILL？
```
场景1：需要实现定时任务功能
场景2：需要管理后台配置任务
场景3：需要任务执行日志可查
```

### 执行流程

```
Step 1: 引入依赖
  - 添加 xxl-job-core
  - 配置 executor

Step 2: 配置执行器
  - 注册到调度中心
  - 配置机器地址

Step 3: 编写任务
  - @XxlJob注解
  - 执行业务逻辑

Step 4: 验证
  - 后台触发执行
  - 检查日志
```

### 配方/素材

**技术栈**：Java 8+, Spring Boot 2.7+
**依赖包**：xxl-job-core
**配置文件**：
```yaml
xxl:
  job:
    executor:
      port: 9999
    handler:
      names: demoJob, syncJob
```

### 脚本

**启动脚本**：
```bash
java -jar xxl-job-executor.jar
```

**测试脚本**：
```bash
curl -X POST http://localhost:8080/run -d "jobId=1"
```

**验证脚本**：
```sql
SELECT * FROM xxl_job_log WHERE job_id=1 ORDER BY id DESC LIMIT 10
```

### 验收标准

```
- [ ] 功能正常：任务能正常执行
- [ ] 单元测试通过：核心方法测试通过
- [ ] 性能达标：执行时间<1分钟
- [ ] 日志可查：后台能看到执行日志
- [ ] 文档已更新：README更新
```
