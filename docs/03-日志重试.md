# 03-日志重试

> 定时任务 - 日志记录与重试机制

---

## ① Why - 价值

**背景与痛点**：
- 任务执行失败后没有记录，无法排查问题
- 网络抖动导致通知失败，业务方收不到回调
- 失败后需要手动重试，运维成本高

**收益**：
- 每次执行都有日志记录，可追溯
- 自动重试机制，提高成功率
- 失败告警，及时发现问题

**用户**：
- 后端开发者、运维工程师

---

## ② What - 定义

**一句话定义**：
XXL-Job 执行器内置日志记录和重试机制，确保任务可追溯、失败可恢复。

**核心组成**：
1. **执行日志** - 记录任务执行过程和结果
2. **重试机制** - 失败后按策略自动重试
3. **回调通知** - 异步回调业务系统

**关键术语**：
- logPath - 执行日志存储路径
- logRetentionDays - 日志保留天数
- notifyTimes - 已通知次数
- maxNotifyTimes - 最大通知次数

## ③ How - 思维

**目标**：实现任务执行日志记录和自动重试

**范围**：
- 允许：`yudao-module-pay/notify/`
- 禁止：修改调度中心表结构

**禁止**：
- 禁止修改 XXL-Job 内置日志表
- 禁止改动重试策略配置

**数据模型**：

```java
// 支付通知任务表
PayNotifyTaskDO:
  - id: Long              # 任务ID
  - type: Integer        # 通知类型(1订单2退款3转账)
  - dataId: Long        # 业务数据ID
  - status: Integer      # 状态(0待通知1成功2失败)
  - notifyTimes: Integer # 已通知次数
  - maxNotifyTimes: Integer # 最大通知次数
  - nextNotifyTime: LocalDateTime # 下次通知时间

// 通知日志表
PayNotifyLogDO:
  - id: Long
  - taskId: Long        # 任务ID
  - notifyTimes: Integer # 通知次数
  - status: Integer    # 状态
  - response: String   # 响应内容
  - createdTime: LocalDateTime
```

**核心流程**：
```
任务触发 
  → 查询待通知列表 
  → 逐个执行通知 
  → 记录日志 
  → 成功则结束 
  → 失败则更新下次通知时间 
  → 等待下次调度重试
```

**关键类**：
- `PayNotifyJob` - 定时扫描任务
- `PayNotifyServiceImpl` - 通知执行服务
- `PayNotifyLogMapper` - 日志记录

## ④ Hard - 难点

**难点1：任务重复执行？**
```
场景：多节点部署，同时触发通知同一笔订单
→ 分布式锁（Redis SETNX）加锁，执行前校验 notifyTimes
```

**难点2：通知超时？**
```
场景：业务接口响应慢，导致任务堆积
→ 设置 NOTIFY_TIMEOUT=120秒，超时强制结束
```

**难点3：重试次数用尽？**
```
场景：通知10次仍失败
→ 更新状态为失败，记录错误日志，不再重试
```

**难点4：分布式环境下日志丢失？**
```
场景：服务重启，日志未持久化
→ 每次执行都记录到数据库，不依赖内存
```

## ⑦ Impl - 实现

**通知次数配置**：

```java
// PayNotifyTaskDO.java
public static final int[] NOTIFY_FREQUENCY = {
    1 * 60,        // 1分钟后
    3 * 60,        // 3分钟后
    10 * 60,       // 10分钟后
    30 * 60,       // 30分钟后
    60 * 60,       // 1小时后
    3 * 60 * 60,  // 3小时后
    12 * 60 * 60, // 12小时后
    24 * 60 * 60  // 24小时后
};
// 最大通知次数 = 数组长度 + 1 = 9次
```

**重试逻辑实现**：

```java
public void executeNotify(PayNotifyTaskDO task) {
    // 1. 分布式锁
    notifyLockCoreRedisDAO.lock(task.getId(), timeout, () -> {
        // 2. 校验通知次数，防止并发
        if (!validateNotifyTimes(task)) {
            return;
        }
        // 3. 执行通知
        executeNotify0(task);
    });
}

@Transactional
public void executeNotify0(PayNotifyTaskDO task) {
    // 1. 调用业务接口
    CommonResult<?> result = executeNotifyInvoke(task);
    
    // 2. 处理结果
    Integer newStatus = processNotifyResult(task, result);
    
    // 3. 记录日志
    notifyLogMapper.insert(PayNotifyLogDO.builder()
        .taskId(task.getId())
        .notifyTimes(task.getNotifyTimes() + 1)
        .status(newStatus)
        .response(response)
        .build());
    
    // 4. 更新下次通知时间
    if (newStatus == PayNotifyStatusEnum.WAITING) {
        task.setNextNotifyTime(calculateNextNotifyTime(task));
        task.update();
    }
}
```

**执行日志配置**：

```yaml
xxl:
  job:
    executor:
      logPath: /tmp/xxl-job
      logRetentionDays: 30
```

## ⑧ SKILL - 提炼

**触发条件**：
- 场景1：支付回调需要重试机制
- 场景2：需要查看任务执行日志
- 场景3：任务失败需要告警

**执行流程**：
```
Step 1: 定时任务扫描
  - PayNotifyJob 每分钟执行
  - 查询待通知任务列表

Step 2: 分布式加锁
  - Redis SETNX 加锁
  - 校验 notifyTimes

Step 3: 执行通知
  - HTTP 调用业务接口
  - 记录响应日志

Step 4: 更新状态
  - 成功 → 状态=成功
  - 失败 → 更新下次通知时间
```

**配方**：
- 最大重试次数：9次
- 超时时间：120秒
- 重试间隔：1分/3分/10分/30分/1时/3时/12时/24时

**验证脚本**：
```sql
-- 查看通知日志
SELECT * FROM pay_notify_log WHERE task_id = 1 ORDER BY notify_times DESC;

-- 查看待通知任务
SELECT * FROM pay_notify_task WHERE status = 0;
```

**验收标准**：
- [ ] 通知成功率高（≥95%）
- [ ] 日志完整可查
- [ ] 失败自动重试
- [ ] 分布式环境无重复
