# 14-租户拦截.md

> 本文档基于八步法分析 SaaS 多租户场景下的 MyBatis Plus 租户拦截实现

---

## ① Why - 价值 (为什么)

### 背景与痛点

在 SaaS 多租户系统中，每个租户的数据必须严格隔离。如果开发者在编写 SQL 时忘记手动添加 `WHERE tenant_id = ?` 条件，就会发生数据泄露——A 租户能看到 B 租户的数据，这在金融、医疗等场景是严重的安全事故。

### 收益

- **安全性↑**：自动为所有查询追加租户条件，杜绝人工遗漏
- **开发效率↑**：开发者无需关注租户过滤，专注业务逻辑
- **维护成本↓**：新增表只需继承基类，自动获得租户能力

### 使用者

- 后端开发工程师
- SaaS 平台运维人员

---

## ② What - 定义 (是什么)

### 一句话定义

基于 MyBatis Plus 的 `TenantLineHandler` 拦截器，在数据库层面自动为所有 SQL 追加租户过滤条件，实现无感的租户数据隔离。

### 核心组成

| 组件 | 职责 |
|------|------|
| `TenantDatabaseInterceptor` | 核心拦截器，实现 `TenantLineHandler` 接口 |
| `TenantContextHolder` | 租户上下文 ThreadLocal 存储 |
| `TenantContextWebFilter` | 从请求 Header 解析租户 ID |
| `TenantBaseDO` | 实体基类，包含 `tenantId` 字段 |
| `TenantProperties` | 配置类，支持忽略表/接口 |

### 关键术语

- **TenantLineHandler**：MyBatis Plus 多租户拦截接口
- **TenantContextHolder**：租户上下文持有者，使用 TTL (TransmittableThreadLocal) 支持线程池
- **TenantIgnore**：注解，标记某个实体类忽略租户拦截

## ③ How - 思维 (怎么做)

### 数据模型设计

```java
// 租户基类实体
public abstract class TenantBaseDO extends BaseDO {
    private Long tenantId;  // 多租户编号
}

// 具体业务实体继承
public class UserDO extends TenantBaseDO {
    private String username;
    private String password;
}
```

### 关键流程

```
请求进入 → TenantContextWebFilter 解析 Header 中的 tenant-id 
         → 设置到 TenantContextHolder 
         → MyBatis Plus 执行 SQL 时触发 TenantDatabaseInterceptor
         → 自动追加 WHERE tenant_id = ?
         → 执行SQL → 清理 TenantContextHolder
```

### 核心拦截逻辑

```java
// TenantDatabaseInterceptor.java
@Override
public Expression getTenantId() {
    return new LongValue(TenantContextHolder.getRequiredTenantId());
}

@Override
public boolean ignoreTable(String tableName) {
    // 1. 全局忽略（TenantContextHolder.isIgnore()）
    if (TenantContextHolder.isIgnore()) {
        return true;
    }
    // 2. 配置忽略的表（如系统表 admin_user）
    if (ignoreTables.contains(tableName)) {
        return true;
    }
    // 3. 未继承 TenantBaseDO 的表，默认忽略
    if (!TenantBaseDO.class.isAssignableFrom(entityType)) {
        return true;
    }
    // 4. 有 @TenantIgnore 注解的表
    return entityType.isAnnotationPresent(TenantIgnore.class);
}
```

### 配置项

```yaml
yudao:
  tenant:
    enable: true  # 是否开启多租户
    ignore-tables:  # 忽略多租户的表
      - auth_role_menu
      - auth_user_role
    ignore-urls:  # 忽略租户检查的接口
      - /api/notify/callback
      - /api/pay/callback
```

## ④ Hard - 难点 (挑战)

### 难点1：哪些表需要忽略？

**问题**：并非所有表都需要租户隔离，系统表（如角色、菜单、字典）通常是全局共享的。

**解决方案**：
- 默认所有表忽略租户
- 只有继承了 `TenantBaseDO` 的表才开启租户隔离
- 可通过 `@TenantIgnore` 注解强制忽略

### 难点2：线程池环境下上下文丢失

**问题**：使用线程池时，ThreadLocal 无法传递，子线程看不到父线程的租户 ID。

**解决方案**：使用 Alibaba 的 `TransmittableThreadLocal` (TTL)，支持线程池场景下的上下文传递。

```java
private static final ThreadLocal<Long> TENANT_ID = new TransmittableThreadLocal<>();
```

### 难点3：RPC 服务间调用

**问题**：微服务 A 调用微服务 B 时，B 服务如何获取租户 ID？

**解决方案**：
- 通过 HTTP Header 传递 `tenant-id`
- 服务间调用使用 `TenantRequestInterceptor` 拦截 Feign/Ribbon 请求，自动追加 Header

### 难点4：跨租户访问

**问题**：超级管理员需要查看某个租户的数据，如何实现？

**解决方案**：
- 使用 `visit-tenant-id` Header 指定要访问的租户
- `TenantContextWebFilter` 优先读取 `visit-tenant-id`

### 难点5：主键生成器兼容

**问题**：Oracle 下使用序列生成主键时，会查询 `DUAL` 表，自动拼接 tenant_id 导致报错。

**解决方案**：在 `TenantDatabaseInterceptor` 中将 `DUAL` 表加入忽略列表。

## ⑥ Select - 选型 (选哪个)

### 候选方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| MyBatis Plus TenantLineHandler | 无感集成、自动拦截、社区活跃 | 依赖 MyBatis Plus | 绝大多数 Spring Boot 项目 |
| ShardingSphere 租户拦截 | 功能强大、支持分片 | 重量级、学习曲线高 | 需要分库分表的场景 |
| 手动拦截 (AOP) | 完全可控 | 重复劳动、易遗漏 | 遗留项目改造 |
| 数据库行级安全 | 数据库原生支持 | 各数据库语法不同 | 对安全性极高 |

### 选型理由

选择 **MyBatis Plus TenantLineHandler**，因为：
1. yudao-cloud 本身基于 MyBatis Plus，无额外依赖
2. 自动拦截所有 SQL，开发者无感知
3. 支持 TTL，解决异步场景
4. 配置灵活，支持忽略表/接口
5. 与 Spring Boot Starter 深度集成

## ⑦ Impl - 实现 (细节)

### 核心类清单

| 类 | 路径 | 职责 |
|---|---|---|
| TenantDatabaseInterceptor | `framework/tenant/.../db/` | SQL 拦截，追加 tenant_id |
| TenantContextHolder | `framework/tenant/.../context/` | 租户上下文 ThreadLocal |
| TenantContextWebFilter | `framework/tenant/.../web/` | 从请求解析租户 ID |
| TenantBaseDO | `framework/tenant/.../db/` | 实体基类 |
| TenantProperties | `framework/tenant/.../config/` | 配置属性 |
| TenantIgnore | `framework/tenant/.../aop/` | 忽略租户注解 |

### 关键代码片段

**1. 实体继承**

```java
// 只要继承 TenantBaseDO，自动获得租户能力
@Data
@EqualsAndHashCode(callSuper = true)
public class OrderDO extends TenantBaseDO {
    private String orderNo;
    private BigDecimal amount;
}
```

**2. 忽略特定表**

```java
// 方式一：配置忽略
yudao:
  tenant:
    ignore-tables:
      - system_dict_data

// 方式二：注解忽略
@TenantIgnore
@Data
public class DictDataDO extends BaseDO {
    // 不参与租户过滤
}
```

**3. 编程式忽略**

```java
// 在需要忽略租户的业务中
TenantUtils.ignore(() -> {
    // 该方法内所有 SQL 不追加 tenant_id
    userMapper.selectById(1L);
});
```

### 校验步骤

```
Step 1: 检查请求是否携带 tenant-id
  → 校验点：Header 中存在 tenant-id
  → 验证方法：打印 TenantContextHolder.getTenantId()

Step 2: 检查 SQL 是否追加条件
  → 校验点：SQL 包含 WHERE tenant_id = ?
  → 验证方法：开启 MyBatis SQL 日志

Step 3: 检查忽略表是否生效
  → 校验点：系统表查询无 tenant_id 条件
  → 验证方法：查询 admin_user 表的 SQL 日志

Step 4: 检查线程池传递
  → 校验点：异步任务中能获取租户 ID
  → 验证方法：异步任务中打印租户 ID
```

### 约束条件

- 实体必须继承 `TenantBaseDO` 才能开启租户隔离
- RPC 调用必须传递 `tenant-id` Header
- 定时任务需手动设置租户上下文

## ⑧ SKILL - 提炼 (复用)

### 触发条件

```
场景1：新业务模块需要接入多租户
场景2：排查租户数据泄露问题
场景3：定时任务需要设置租户上下文
场景4：跨租户数据访问
```

### 执行流程

```
Step 1: 确认实体继承 TenantBaseDO
  → 检查实体类是否 extends TenantBaseDO
  
Step 2: 配置忽略表（如需要）
  → 在 yudao.tenant.ignore-tables 添加
  
Step 3: RPC 调用传递租户 ID
  → 确保 Feign Client 传递 tenant-id Header
  
Step 4: 验证
  → 打印 SQL 日志确认 tenant_id 追加
```
