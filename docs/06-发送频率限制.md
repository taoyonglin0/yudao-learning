# 06-发送频率限制

> 发送频率限制是消息通知系统中，防止恶意刷短信/通知的核心防护功能。本文档按照八步法进行分析。

---

## ① Why - 价值 (为什么)

### 背景与痛点
- **业务场景**：短信验证码、通知消息发送可能被恶意调用
- **痛点**：无限制发送导致资源浪费、资金损失、用户体验下降
- **需求**：对发送频率和数量进行限制

### 价值收益
- ✅ **防刷保护**：防止恶意批量发送
- ✅ **成本控制**：限制发送量，节省费用
- ✅ **用户体验**：避免用户收到骚扰短信
- ✅ **合规要求**：满足运营商监管要求

### 使用者
- 验证码发送场景
- 通知消息发送场景
- 营销消息发送场景

---

## ② What - 定义 (是什么)

### 核心概念
**发送频率限制**：对用户发送消息的频率和数量进行控制，防止恶意刷发送。

### 核心组成部分
1. **SmsCodeProperties** - 频率限制配置
2. **createSmsCode()** - 发送频率校验
3. **@RateLimiter** - 分布式限流注解
4. **RateLimiterService** - 限流服务

### 关键术语
| 术语 | 说明 |
|------|------|
| 发送频率 | 两次发送的最小间隔时间 |
| 每日上限 | 每天允许发送的最大数量 |
| 限流器 | 用于分布式场景的令牌桶/滑动窗口 |
| IP 限流 | 按用户IP维度限制 |

### 关键代码设计

#### 1. 频率校验
```java
private String createSmsCode(String mobile, Integer scene, String ip) {
    SmsCodeDO lastSmsCode = smsCodeMapper.selectLastByMobile(mobile, null, null);
    if (lastSmsCode != null) {
        // 校验发送频率
        if (LocalDateTimeUtil.between(lastSmsCode.getCreateTime(), LocalDateTime.now()).toMillis()
                < smsCodeProperties.getSendFrequency().toMillis()) {
            throw exception(SMS_CODE_SEND_TOO_FAST);
        }
        // 校验每日上限
        if (isToday(lastSmsCode.getCreateTime()) && 
                lastSmsCode.getTodayIndex() >= smsCodeProperties.getSendMaximumQuantityPerDay()) {
            throw exception(SMS_CODE_EXCEED_SEND_MAXIMUM_QUANTITY_PER_DAY);
        }
    }
    // 创建验证码
    String code = String.format("%06d", randomInt(100000, 999999));
    // ...
}
```

// 使用 Redis 实现滑动窗口限流
public boolean acquire(String key, int count, int time) {
    String luaScript = """
        local key = KEYS[1]
        local now = tonumber(ARGV[1])
        local window = tonumber(ARGV[2])
        local limit = tonumber(ARGV[3])
        
        redis.call('zremrangebyscore', key, 0, now - window)
        local current = redis.call('zcard', key)
        if current < limit then
            redis.call('zadd', key, now, now .. math.random())
            redis.call('expire', key, window)
            return 1
        end
        return 0
        """;
    // 执行 Lua 脚本
}

## ④ Hard - 难点 (挑战)

### 难点1：时区问题
```java
// 场景：跨时区的每日计数
// 难点：如何定义"今天"的开始时间
private boolean isToday(LocalDateTime createTime) {
    // 使用服务器本地时区判断
    return createTime.toLocalDate().equals(LocalDate.now());
}
// 注意：服务器时区应与业务时区一致
```

### 难点2：分布式限流
```java
// 场景：多台服务器部署
// 难点：需要跨服务的统一限流
// 解决方案：使用 Redis 存储计数
// Redis key: rate_limit:{mobile}:{window}
// 使用 Lua 脚本保证原子性
```

### 难点3：IP 限流绕过
```java
// 场景：用户更换IP绕过限制
// 难点：代理IP获取不可靠
// TODO: 实现每个IP每天可发送数量限制
// TODO: 实现每个IP每小时可发送数量限制
// 注意：需要考虑内网IP场景
```

### 难点4：计数器重置
```java
// 场景：计数器跨天重置
// 难点：需要在每天0点清零
// 解决方案：
// 1. 每次创建时检查日期，非今天则重置
// 2. 定时任务每天0点执行重置
todayIndex(lastSmsCode != null && isToday(lastSmsCode.getCreateTime()) 
    ? lastSmsCode.getTodayIndex() + 1 
    : 1)
```

## ⑥ Select - 选��个

### 候选方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 内存限流 | 性能高 | 不支持分布式 | 单机场景 |
| Redis 限流 | 支持分布式 | 有网络开销 | 分布式场景 |
| 数据库限流 | 数据可靠 | 性能低 | 简单场景 |
| 限流服务 | 功能全面 | 需要额外部署 | 大型系统 |

### 项目选型
**yudao-cloud 选择 Redis + 数据库限流**，因为：
1. Redis 用于 API 限流 (@RateLimiter)
2. 数据库用于验证码发送频率记录
3. 组合使用，满足不同场景
4. 无需额外部署

## ⑦ Impl - 实现 (细节)

### 核心配置类

#### SmsCodeProperties
```java
@ConfigurationProperties(prefix = "yudao.sms-code")
@Data
public class SmsCodeProperties {
    private Duration expireTimes;        // 10m
    private Duration sendFrequency;        // 60s
    private Integer sendMaximumQuantityPerDay;  // 10
    private Integer beginCode;           // 100000
    private Integer endCode;           // 999999
}
```

### 限流注解

#### @RateLimiter
```java
@Target({ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
public @interface RateLimiter {
    int time() default 1;           // 时间值
    TimeUnit timeUnit() default TimeUnit.SECONDS;
    int count() default 100;         // 允许次数
    String message() default "";
    Class<? extends RateLimiterKeyResolver> keyResolver() default DefaultRateLimiterKeyResolver.class;
    String keyArg() default "";
}
```

### Key 解析器

| 类型 | 说明 | 示例 |
|------|------|------|
| DefaultRateLimiterKeyResolver | 全局级别 | 限流所有请求 |
| UserRateLimiterKeyResolver | 用户ID级别 | 每个用户限流 |
| ClientIpRateLimiterKeyResolver | IP级别 | 每个IP限流 |
| ServerNodeRateLimiterKeyResolver | 服务器级别 | 每台服务器限流 |
| ExpressionRateLimiterKeyResolver | 自定义表达式 | 按表达式计算 |

### 核心流程步骤

```
Step 1: 获取配置
  → 校验点：配置有效
  → 验证方法：检查 sendFrequency

Step 2: 查询上次记录
  → 校验点：lastSmsCode != null
  → 验证方法：selectLastByMobile

Step 3: 频率校验
  → 校验点：时间差 >= sendFrequency
  → 验证方法：LocalDateTimeUtil.between

Step 4: 每日上限校验
  → 校验点：todayIndex < maxPerDay
  → 验证方法：检查 todayIndex

Step 5: 创建记录
  → 校验点：todayIndex + 1
  → 验证方法：insert成功
```

### 失败恢复机制

```
场景1：发送过快
  → 触发：时间差 < sendFrequency
  → 恢复：抛出 SMS_CODE_SEND_TOO_FAST
  → 提示："发送过快，请 {sendFrequency} 后重试"

场景2：超过每日上限
  → 触发：todayIndex >= maxPerDay
  → 恢复：抛出 SMS_CODE_EXCEED_SEND_MAXIMUM_QUANTITY_PER_DAY
  → 提示："今日发送次数已达上限"

场景3：验证码过期
  → 触发：时间差 > expireTimes
  → 恢复：抛出 SMS_CODE_EXPIRED

场景4：验证码已使用
  → 触发：used == true
  → 恢复：抛出 SMS_CODE_USED
```
