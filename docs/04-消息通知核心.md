# 04-消息通知核心

> 消息通知模块是 yudao-cloud 系统中，实现站内信（通知消息）发送与管理的核心功能。本文档按照八步法进行分析。

---

## ① Why - 价值 (为什么)

### 背景与痛点
- **业务场景**：系统需要向用户发送通知消息，如订单状态变更、系统公告、验证码通知等
- **痛点**：没有统一的通知模块，每个业务单独实现导致代码重复、维护困难
- **需求**：统一的消息通知平台，支持模板化发送，支持站内信场景

### 价值收益
- ✅ **效率提升**：模板化参数替换，发送逻辑复用
- ✅ **体验提升**：用户可在系统内查看历史消息，未读标记
- ✅ **维护性**：统一管理通知模板，修改模板即时生效

### 使用者
- 管理员（ ADMIN ）
- 会员用户（ MEMBER ）

---

## ② What - 定义 (是什么)

### 核心概念
**消息通知模块**：站内信发送与管理系统，提供通知模板管理和消息发送能力。

### 核心组成部分
1. **NotifyTemplateDO** - 通知模板
2. **NotifyMessageDO** - 通知消息
3. **NotifySendService** - 消息发送服务
4. **NotifyMessageService** - 消息管理服务
5. **NotifyTemplateService** - 模板管理服务

### 关键术语
| 术语 | 说明 |
|------|------|
| 站内信 | 系统内的通知消息，类似邮件 |
| 模板 | 预先配置的通知内容，含参数占位符 |
| 已读/未读 | 消息阅读状态 |
| 参数替换 | 模板中 {name} 替换为实际值 |

## ④ Hard - 难点 (挑战)

### 难点1：模板参数校验
```java
// 场景：模板定义了参数，但调用方没传
// 难点：如何优雅校验必填参数
public void validateTemplateParams(NotifyTemplateDO template, Map<String, Object> templateParams) {
    template.getParams().forEach(key -> {
        Object value = templateParams.get(key);
        if (value == null) {
            throw exception(NOTIFY_SEND_TEMPLATE_PARAM_MISS, key);
        }
    });
}
```

### 难点2：模板内容格式化
```java
// 场景：模板内容 "您好 {name}，您的订单 {orderNo} 已发货"
// 难点：安全替换，防止注入
public String formatNotifyTemplateContent(String content, Map<String, Object> params) {
    for (Map.Entry<String, Object> entry : params.entrySet()) {
        content = content.replace("{" + entry.getKey() + "}", String.valueOf(entry.getValue()));
    }
    return content;
}
```

## ⑥ Select - 选型 (选哪个)

### 候选方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 自研站内信 | 完全可控、与业务深度集成 | 需要开发 | 企业内部系统 |
| 极光推送 | 覆盖APP推送、邮件 | 需付费、有依赖 | 移动APP为主 |
| 短信服务商 | 覆盖短信、验证码 | 需付费、通道管理 | 验证码场景 |

### 项目选型
**yudao-cloud 选择自研站内信**，因为：
1. 与项目用户体系深度集成（管理员+会员）
2. 模板化设计满足业务运营需求
3. 支持历史消息存储与查询
4. 无外部依赖，部署简单

### 核心流程步骤

```
Step 1: 校验模板存在
  → 校验点：template != null
  → 验证方法：查询数据库

Step 2: 校验模板启用
  → 校验点：status == 1
  → 验证方法：检查状态

Step 3: 校验参数完整
  → 校验点：params全部传入
  → 验证方法：forEach校验

Step 4: 内容替换
  → 校验点：占位符替换完成
  → 验证方法：返回替换后内容

Step 5: 创建消息
  → 校验点：insert成功
  → 验证方法：返回消息ID
```

### 失败恢复机制

```
场景1：模板不存在
  → 触发：getNotifyTemplateByCodeFromCache返回null
  → 恢复：抛出NOTICE_NOT_FOUND异常

场景2：参数缺失
  → 触发：templateParams.get(key) == null
  → 恢复：抛出NOTIFY_SEND_TEMPLATE_PARAM_MISS异常

场景3：模板已禁用
  → 触发：status == 0
  → 恢复：返回null，记录日志
```
