# 82-流控规则

> 服务保护 - 流控规则核心原理与实现

---

## ① Why - 价值

**背景与痛点**：
- 秒杀活动时大量请求涌入，数据库连接池耗尽
- 核心接口被刷，响应变慢，影响正常用户
- 没有流控规则，业务系统毫无保护

**收益**：
- 保护核心接口可用性
- 削峰填谷，减轻下游压力
- 快速失败，避免雪崩

**用户**：
- 网关层、服务层、数据库层

---

## ② What - 定义

**一句话定义**：
流控规则是一种流量控制手段，通过限制并发/速率保护系统，可用BDRateLimiter实现。

**核心组成**：
1. **限流类型** - 并发数限制、速率限制
2. **限流粒度** - 全局、用户、IP、接口
3. **限流策略** - 拒绝、降级、排队

**关键术语**：
- RRateLimiter - Redisson分布式限流器
- RateType.OVERALL - 全局限流
- RateType.PER_CLIENT - 单客户端限流

## ③ How - 思维

**目标**：实现分布式流控能力

**范围**：
- 允许：yudao-framework/yudao-spring-boot-starter-protection/
- 禁止：改动业务模块核心逻辑

**限流模型**：

```java
// 流控规则配置
public class FlowControlProperties {
    private Integer count = 100;      // 限流次数
    private Integer time = 1;         // 时间窗口(秒)
    private TimeUnit timeUnit = TimeUnit.SECONDS;
    private RateType rateType = RateType.OVERALL;
}
```

**核心流程**：
```
请求进入 
  → 获取限流Key 
  → 尝试获取令牌 
  → 成功→执行业务 
  → 失败→返回限流提示
```

## ④ Hard - 难点

**难点1：限流粒度如何选择？**
```
场景：全局限流vs用户级限流vs接口级限流
→ yudao提供5种Key Resolver任选
```

**难点2：限流阈值如何配置？**
```
场景：阈值太小影响正常用户，阈值太大起不到保护
→ 需要压测确定合理阈值
```

**难点3：分布式限流一致性？**
```
场景：多节点如何保证全局限流
→ 使用Redis RRateLimiter保证一致性
```

## ⑥ Select - 选型

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| Redisson | 分布式、性能高 | 需要Redis | 高并发场景 |
| Sentinel | 功能全，阿里出品 | 较重 | 阿里技术栈 |
| bucket4j | Java简单 | 无分布式 | 单机场景 |
| Nginx | 网关限流 | 需配置Lua | 入口限流 |

**选型理由**：
1. 项目已有Redisson，开箱即用
2. RRateLimiter性能优秀
3. 支持PER_CLIENT全局和单客户端模式

## ⑦ Impl - 实现

**核心实现**：

```java
// RateLimiterRedisDAO 核心方法
public Boolean tryAcquire(String key, int count, int time, TimeUnit timeUnit) {
    RRateLimiter rateLimiter = getRRateLimiter(key, count, time, timeUnit);
    return rateLimiter.tryAcquire();
}
```

```java
// RateLimiterAspect 切面
@Before("@annotation(rateLimiter)")
public void beforePointCut(JoinPoint joinPoint, RateLimiter rateLimiter) {
    String key = keyResolver.resolver(joinPoint, rateLimiter);
    boolean success = rateLimiterRedisDAO.tryAcquire(key,
            rateLimiter.count(), rateLimiter.time(), rateLimiter.timeUnit());
    if (!success) {
        throw new ServiceException(TOO_MANY_REQUESTS);
    }
}
```

**使用示例**：

```java
// 接口级限流：每秒1000次
@RateLimiter(time = 1, count = 1000)
public void list() { }

// 用户级限流：每个用户每秒100次
@RateLimiter(time = 1, count = 100, keyResolver = UserRateLimiterKeyResolver.class)
public void query() { }

// IP级限流：每个IP每秒50次
@RateLimiter(time = 1, count = 50, keyResolver = ClientIpRateLimiterKeyResolver.class)
public void submit() { }
```

**配置示例**：

```yaml
# application.yml
rate-limiter:
  default:
    count: 100
    time: 1
```

## ⑧ SKILL - 提炼

**触发条件**：
- 场景1：需要保护核心接口不被刷爆
- 场景2：需要限制用户请求频率
- 场景3：需要削峰填谷保护下游

**执行流程**：
```
Step 1: 确认已有protection依赖
Step 2: 在需要限流的方法上添加注解
Step 3: 配置限流参数(count/time/keyResolver)
Step 4: 验证限流效果
```

**配方**：
- 技术栈：Java 8+, Spring Boot 2.7+, Redisson
- 依赖：yudao-spring-boot-starter-protection
- 注解：@RateLimiter

**验收标准**：
- [ ] 限流注解生效：超过阈值正确拦截
- [ ] 正常请求正常：阈值内请求通过
- [ ] 性能达标：限流开销<5ms
- [ ] 日志可查：拦截有记录
