# 24-表单验证 - 动态表单之验证规则

> 本文档按照八步法（Why-What-How-Hard-Metric-Select-Impl-SKILL）分析动态表单验证规则技术

---

## ① Why - 价值 (为什么)

### 背景与痛点

在 Web 应用开发中，表单验证是保障数据质量的第一道防线：

1. **数据质量**：用户输入可能存在格式错误、范围超限、必填项缺失等问题
2. **安全防护**：恶意用户可能提交畸形数据攻击系统
3. **用户体验**：及时反馈错误，避免提交后才告知问题
4. **后端压力**：减少无效请求对后端的压力

### 带来的价值

- ✅ **数据准确性**：确保入库数据符合业务规则
- ✅ **安全性**：防止 SQL 注入、XSS 等安全问题
- ✅ **用户体验**：前端即时校验，响应迅速
- ✅ **开发效率**：声明式配置，无需重复编写校验代码

### 使用者

- 后端开发工程师
- 前端开发工程师
- 需要构建表单系统的业务场景

## ② What - 定义 (是什么)

### 一句话定义

**表单验证规则**是指通过声明式配置（注解），在数据进入系统前对字段的格式、范围、必填性等进行校验的机制。

### 核心组成部分

| 组件 | 说明 |
|------|------|
| **JSR-303/JSR-380** | Java 官方校验规范（hibernate-validator 实现） |
| **@Valid / @Validated** | 触发校验的注解 |
| **ConstraintViolation** | 校验违反异常 |
| **GlobalExceptionHandler** | 全局异常处理，统一返回错误信息 |
| **Swagger/OAS 注解** | API 文档中的字段说明 |

### 关键术语

- **Bean Validation**：Java EE 校验规范
- **Constraint**：约束条件
- **ConstraintViolation**：约束违反
- **Message**：错误消息模板

## ③ How - 思维 (怎么做)

### 常用校验注解

| 注解 | 说明 | 示例 |
|------|------|------|
| `@NotNull` | 不能为 null | 用户ID |
| `@NotEmpty` | 不能为空（字符串/集合） | 用户名 |
| `@NotBlank` | 不能为空白字符串 | 密码 |
| `@Size` | 长度范围 | @Size(min=2, max=20) |
| `@Length` | 字符串长度 | @Length(min=2, max=50) |
| `@Min` / `@Max` | 数值范围 | @Min(0) @Max(100) |
| `@Range` | 数值范围 | @Range(min=0, max=100) |
| `@Pattern` | 正则表达式 | @Pattern(regexp="^1[3-9]\\d{9}$") |
| `@Email` | 邮箱格式 | @Email |
| `@URL` | URL 格式 | @URL |
| `@DecimalMin/Max` | BigDecimal 范围 | @DecimalMin("0.01") |
| `@Digits` | 数字格式 | @Digits(integer=10, fraction=2) |
| `@Past` / `@Future` | 时间范围 | @Past / @Future |
| `@Valid` | 嵌套对象校验 | 嵌套 VO |

## ④ Hard - 难点 (挑战)

### 难点 1：嵌套对象校验

```java
// 嵌套对象需要 @Valid 触发校验
public class OrderSaveReqVO {
    @NotNull
    private Long id;
    
    @Valid  // 必须加，否则嵌套对象不校验
    @NotNull
    private UserVO user;
}
```

### 难点 2：分组校验

同一个 VO 在不同场景下校验规则不同：

```java
public class UserVO {
    @NotNull(groups = {Update.class})  // 更新时校验
    private Long id;
    
    @NotBlank(groups = {Create.class, Update.class})  // 都校验
    private String name;
}
```

### 难点 3：自定义校验

当内置注解不满足需求时，需要自定义：

```java
// 1. 创建注解
@Constraint(validatedBy = PhoneValidator.class)
@Target({METHOD, FIELD})
@Retention(RUNTIME)
public @interface IsPhone {
    String message() default "手机号格式错误";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}

// 2. 实现校验器
public class PhoneValidator implements ConstraintValidator<IsPhone, String> {
    @Override
    public boolean isValid(String value, ConstraintValidatorContext context) {
        return value == null || value.matches("^1[3-9]\\d{9}$");
    }
}
```

### 难点 4：前端校验 vs 后端校验

- 前端校验：提升用户体验，但可被绕过
- 后端校验：安全保障，必须有
- **最佳实践**：前后端都要做，后端是最后防线

## ⑥ Select - 选型 (选哪个)

### 候选方案

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **Hibernate Validator** | 规范实现、功能丰富 | 学习曲线 | 企业级应用 |
| Spring Validation | Spring 生态集成 | 依赖 Hibernate | Spring 项目 |
| 自研校验框架 | 完全可控 | 工作量大 | 特殊需求 |

### 选型理由

选择 **Hibernate Validator + Spring Validation**，因为：

1. ✅ **标准规范**：JSR-303/JSR-380 标准
2. ✅ **Spring 集成**：开箱即用
3. ✅ **功能丰富**：内置 20+ 注解
4. ✅ **与 yudao-cloud 技术栈一致**：项目本身基于 Spring Boot

### 核心代码 - VO 定义

### 核心代码 - GlobalExceptionHandler

```java
/**
 * 处理 MethodArgumentNotValidException（@RequestBody 校验失败）
 */
@ExceptionHandler(MethodArgumentNotValidException.class)
public CommonResult<?> methodArgumentNotValidExceptionHandler(MethodArgumentNotValidException ex) {
    FieldError fieldError = ex.getBindingResult().getFieldError();
    return error(
        VALIDATION_ERROR.getCode(),
        fieldError != null ? fieldError.getDefaultMessage() : VALIDATION_ERROR.getMsg()
    );
}

/**
 * 处理 ConstraintViolationException（单个参数校验失败）
 */
@ExceptionHandler(ConstraintViolationException.class)
public CommonResult<?> constraintViolationExceptionHandler(ConstraintViolationException ex) {
    ConstraintViolation<?> violation = ex.getConstraintViolations().iterator().next();
    return error(
        VALIDATION_ERROR.getCode(),
        violation.getMessage()
    );
}

/**
 * 处理 BindException（表单绑定失败）
 */
@ExceptionHandler(BindException.class)
public CommonResult<?> bindExceptionHandler(BindException ex) {
    FieldError fieldError = ex.getBindingResult().getFieldError();
    return error(
        VALIDATION_ERROR.getCode(),
        fieldError != null ? fieldError.getDefaultMessage() : VALIDATION_ERROR.getMsg()
    );
}
```

### 校验流程

```
用户提交 → Controller 接收 → @Valid 触发校验
                              ↓
                    ┌─────────┴─────────┐
                    ↓                   ↓
              校验通过              校验失败
                    ↓                   ↓
              执行业务逻辑      GlobalExceptionHandler 捕获
                              ↓
                    返回错误信息给前端
```

## ⑧ SKILL - 提炼 (复用)

### 触发条件

- 场景 1：需要校验用户提交的表单数据
- 场景 2：需要统一处理校验错误
- 场景 3：需要自定义校验规则

### 执行流程

```
Step 1: 添加依赖
  - Spring Boot 项目已内置 hibernate-validator

Step 2: 定义 VO
  - 在 VO 类上添加校验注解
  - 使用 @Valid 触发嵌套对象校验

Step 3: Controller 使用
  - 在方法参数上添加 @Valid 注解

Step 4: 异常处理
  - GlobalExceptionHandler 已处理，无需额外配置
```

