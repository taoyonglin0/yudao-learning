## ① Why - 价值 (为什么)

### 背景与痛点

在企业管理系统中，不同角色能看到的数据范围是不同的：

| 角色 | 能看到的数据 |
|------|-------------|
| 超级管理员 | 所有数据 |
| 部门经理 | 本部门数据 |
| 组长 | 本组数据 |
| 普通员工 | 仅自己创建的数据 |

**不用数据权限控制有什么痛点？**

1. **数据泄露**
   - 场景：员工A能看到同事B的客户名单
   - 结果：商业机密泄露，违反合规要求

2. **权限管理混乱**
   - 场景：每个接口都手动判断数据归属
   - 结果：代码重复、难以维护、容易遗漏

3. **无法满足业务需求**
   - 场景：需要按部门/区域/项目隔离数据
   - 结果：无法实现，业务受限

**用了数据权限控制能带来什么收益？**

| 收益 | 说明 |
|------|------|
| ✅ 数据安全 | 按角色自动过滤数据，防止泄露 |
| ✅ 代码复用 | 一次配置，全局生效 |
| ✅ 灵活配置 | 支持多种权限级别（全部/部门/仅自己） |
| ✅ 审计合规 | 数据访问可追溯 |

## ② What - 定义 (是什么)

### 一句话定义

**数据权限控制**是一种基于角色的数据访问控制机制，在数据库查询时自动注入过滤条件，确保用户只能看到有权限的数据。

### 核心组成部分

```
数据权限 = 权限规则定义 + 权限级别 + SQL自动拦截 + 动态配置
```

1. **权限规则定义**：配置哪些表需要数据权限控制
2. **权限级别**：全部、本部门、仅自己等
3. **SQL自动拦截**：通过 AOP/Plugin 自动添加 WHERE 条件
4. **动态配置**：管理员可在界面配置权限规则

### 关键术语

| 术语 | 说明 |
|------|------|
| Data Permission | 数据权限，控制用户可见数据范围 |
| Dept Data Scope | 部门数据范围，定义可访问的部门 |
| Row-Level Security | 行级安全，基于行的数据访问控制 |
| Tenant Isolation | 租户隔离，多租户场景的数据隔离 |

## ③ How - 思维 (怎么做)

### 如果我们自己实现，怎么做？

#### 方案一：代码手动判断（不推荐）

```java
public List<Order> getOrderList(Long userId) {
    User user = getUser(userId);
    
    // 手动判断权限
    if (user.isAdmin()) {
        return orderMapper.selectAll();
    } else if (user.isDeptManager()) {
        return orderMapper.selectByDeptId(user.getDeptId());
    } else {
        return orderMapper.selectByUserId(userId);
    }
}
```

**缺点**：代码重复、难以维护、容易遗漏

#### 方案二：SQL 拦截器（推荐）

```java
@Intercepts({
    @Signature(type = Executor.class, method = "query", 
               args = {MappedStatement.class, Object.class, RowBounds.class, ResultHandler.class})
})
public class DataPermissionInterceptor implements Interceptor {
    
    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        // 1. 获取当前用户权限
        DataPermissionContext context = DataPermissionContextHolder.getContext();
        
        // 2. 构建过滤 SQL
        String whereSql = buildWhereSql(context);
        
        // 3. 修改 SQL，注入 WHERE 条件
        MappedStatement ms = invocation.getArgs()[0];
        BoundSql boundSql = ms.getBoundSql(invocation.getArgs()[1]);
        String newSql = boundSql.getSql() + " AND " + whereSql;
        
        // 4. 执行修改后的 SQL
        // ...
    }
}
```

## ④ Hard - 难点 (挑战)

### 可能遇到的问题

**难点1：子部门数据可见**
```
场景：部门经理要看到本部门及所有子部门的数据
→ 解决：递归查询子部门，构建 dept_id IN (1, 2, 3, ...) 条件
```

**难点2：跨部门协作**
```
场景：项目A需要跨部门3人协作，如何让他们互相看到数据？
→ 解决：使用"自定义数据权限"，手动配置可访问的用户/部门
```

**难点3：关联表查询**
```
场景：订单表没有 dept_id，如何按部门过滤？
→ 解决：关联用户表，通过用户.dept_id 过滤
```

**难点4：动态 SQL**
```
场景：有些查询是动态构建的，拦截器如何处理？
→ 解决：使用 MyBatis StatementHandler 拦截，在编译前修改
```

**难点5：性能问题**
```
场景：数据权限过滤导致查询变慢
→ 解决：建立索引、优化 SQL、缓存权限数据
```

### 边界情况

| 边界情况 | 处理方式 |
|----------|----------|
| 用户无部门 | 默认"仅自己" |
| 管理员 | 跳过权限过滤 |
| 租户隔离 | 优先租户过滤，再数据权限 |
| 脱敏字段 | 在 Java 层处理 |

## ⑥ Select - 选型 (选哪个)

### 候选方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **MyBatis 拦截器** | 无侵入、性能好 | 需要熟悉 MyBatis | 大多数场景 |
| Spring AOP | 简单易用 | 只拦截 Service | 简单场景 |
| 数据库视图 | 性能好 | 维护复杂 | 报表场景 |
| 行级安全策略 | 数据库层实现 | 特定数据库 | 高安全场景 |

### 为什么选 MyBatis 拦截器？

```
最后选 MyBatis 拦截器，因为：
1. 在 SQL 执行前拦截，无业务代码侵入
2. 性能影响小，直接修改 SQL
3. yudao-cloud 已有成熟实现
4. 支持复杂关联查询
5. 与 MyBatis Plus 完美集成
```

**核心组件**：

| 组件 | 路径 | 说明 |
|------|------|------|
| DataPermissionConfiguration | system/framework/datapermission/config | 系统模块配置 |
| DataPermissionRuleHandler | framework/datapermission/core/db | 权限规则处理器 |
| DataPermissionAnnotationInterceptor | framework/datapermission/core/aop | 注解拦截器 |
| DataPermissionContextHolder | framework/datapermission/core/aop | 上下文持有者 |

**配置类**：

```java
@Configuration
public class DataPermissionConfiguration {

    @Bean
    public DeptDataPermissionRuleCustomizer sysDeptDataPermissionRuleCustomizer() {
        return rule -> {
            // 配置哪些表需要数据权限
            rule.addDeptColumn(AdminUserDO.class);     // 用户表按部门过滤
            rule.addDeptColumn(DeptDO.class, "id");    // 部门表按ID过滤
            rule.addUserColumn(AdminUserDO.class, "id"); // 按用户ID过滤
        };
    }
}
```

### 权限规则定义

```java
// 部门数据权限规则
public class DeptDataPermissionRule implements DataPermissionRule {
    
    @Override
    public String getTableName() {
        return "system_user"; // 用户表
    }
    
    @Override
    public String getDeptColumnName() {
        return "dept_id"; // 部门ID字段
    }
    
    @Override
    public String getUserColumnName() {
        return "id"; // 用户ID字段
    }
}
```

### 数据权限工具类

```java
// 获取当前用户的数据权限范围
@DataPermission
public List<UserDO> getUserList() {
    // 无需手动判断，框架自动处理
    return userMapper.selectList();
}

// 禁用数据权限（特殊场景）
DataPermissionUtils.executeIgnore(() -> {
    // 这里的查询不会添加数据权限过滤
    adminUserMapper.selectById(1L);
});
```

### 执行流程

```
Step 1: 确定需要权限控制的表
  - 分析业务表结构
  - 识别数据归属字段（dept_id, user_id）

Step 2: 配置数据权限规则
  - 在 DataPermissionConfiguration 中添加规则
  - 指定部门字段和用户字段

Step 3: 配置用户权限级别
  - 在用户管理中设置数据范围
  - 部门经理 → 本部门
  - 普通员工 → 仅自己

Step 4: 测试验证
  - 不同角色登录测试
  - 验证数据过滤效果
```

### 配置模板

```java
// 数据权限配置示例
@Configuration
public class MyDataPermissionConfig {

    @Bean
    public DeptDataPermissionRuleCustomizer customDataPermission() {
        return rule -> {
            // 订单表按部门过滤
            rule.addDeptColumn(OrderDO.class, "dept_id");
            // 客户表按负责人过滤
            rule.addUserColumn(CustomerDO.class, "owner_user_id");
        };
    }
}
```

### 代码模板

```java
/**
 * 数据权限使用示例
 * 
 * 1. Service 层添加 @DataPermission 注解
 * 2. 框架自动注入权限过滤
 */
@DataPermission(deptId = "deptId", userId = "id")
public List<OrderDO> getOrderList(OrderQuery query) {
    return orderMapper.selectList(query);
}

/**
 * 禁用数据权限（管理员查看所有数据）
 */
public void exportAllOrders() {
    DataPermissionUtils.executeIgnore(() -> {
        // 导出全部订单，不受权限控制
        export(orderMapper.selectAll());
    });
}
```
