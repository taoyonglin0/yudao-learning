## ① Why - 价值 (为什么)

### 背景与痛点

在分布式系统中，存在一类特殊的请求：**热点参数请求**。所谓热点参数，是指那些具有**高访问频率**或**突发流量**的参数值。

**不用热点参数保护有什么痛点？**

1. **数据库被打挂**
   - 场景：某商品 ID=1000 是爆款，1万用户同时请求 `GET /product/1000`
   - 结果：数据库瞬间 10000 次查询，CPU 飙升，服务不可用

2. **缓存击穿**
   - 场景：热点 key 过期瞬间，大量请求同时穿透到数据库
   - 结果：数据库压力暴增

3. **资源被少数用户霸占**
   - 场景：某用户 ID=8888 是 VIP，频繁请求接口
   - 结果：其他用户等待时间长，体验差

4. **热点数据导致集群故障**
   - 场景：热点参数每次都路由到同一台机器
   - 结果：这台机器先挂，然后雪崩

**用了热点参数保护能带来什么收益？**

| 收益 | 说明 |
|------|------|
| ✅ 系统稳定性 | 防止热点数据压垮数据库 |
| ✅ 资源公平 | 避免少数用户霸占资源 |
| ✅ 故障隔离 | 热点请求被限流，不影响其他用户 |
| ✅ 成本节约 | 减少服务器扩容需求 |

## ② What - 定义 (是什么)

### 一句话定义

**热点参数保护**是一种针对特殊参数值进行流量控制的机制，对高频访问的参数值进行限流，防止热点数据打垮系统。

### 核心组成部分

```
热点参数保护 = 参数提取 + 热点识别 + 限流策略 + 熔断降级
```

1. **参数提取**：从请求中提取需要保护的参数（如 userId、productId）
2. **热点识别**：统计参数值的访问频率，识别出热点
3. **限流策略**：对热点参数值单独限流
4. **熔断降级**：超过阈值时快速失败，返回默认值

### 关键术语

| 术语 | 说明 |
|------|------|
| Hot Parameter | 热点参数，访问频率高的参数值 |
| Parameter Flow Control | 参数级限流，精确到参数值的限流 |
| Sentinel | 阿里巴巴开源的流量控制组件，支持热点参数 |
| Flow Rule | 流量规则，控制请求速率 |
| Degrade Rule | 熔断规则，系统异常时快速失败 |

## ③ How - 思维 (怎么做)

### 如果我们自己实现，怎么做？

#### 方案一：Redis 计数器实现

```java
@Service
public class HotParamService {

    @Autowired
    private StringRedisTemplate redisTemplate;

    /**
     * 热点参数限流
     * @param paramValue 参数值（如商品ID）
     * @param threshold 阈值
     * @return 是否允许通过
     */
    public boolean tryAcquire(String paramValue, int threshold) {
        String key = "hot:param:" + paramValue;
        Long count = redisTemplate.opsForValue().increment(key);
        if (count == 1) {
            // 设置过期时间，统计窗口为1秒
            redisTemplate.expire(key, 1, TimeUnit.SECONDS);
        }
        return count <= threshold;
    }
}
```

#### 方案二：集成 Sentinel（生产推荐）

```java
// Sentinel 热点参数限流配置
ParamFlowRule rule = new ParamFlowRule("getProduct")
    .setParamIdx(0)  // 第一个参数（商品ID）
    .setCount(100)   // 单机阈值 100QPS
    .setControlBehavior(BLOCK_PASS)
    .setMaxQueueingTimeMs(500);  // 最大排队时间

ParamFlowRuleManager.loadRules(Collections.singletonList(rule));
```

### 数据模型设计

**热点参数规则表 (可选)**：
```sql
CREATE TABLE hot_param_rule (

    resource        VARCHAR(100) NOT NULL COMMENT '资源名',
    param_idx       INT NOT NULL COMMENT '参数索引',
    param_value     VARCHAR(200) COMMENT '参数值，NULL表示所有',
    count           INT NOT NULL COMMENT '限流阈值',
    time_window     INT DEFAULT 1 COMMENT '时间窗口(秒)',
    status          TINYINT DEFAULT 1 COMMENT '状态 0禁用 1启用',
    created_time    DATETIME,
    updated_time    DATETIME

```

### 关键流程设计

```
请求进入 → 提取参数 → 判断是否热点 → 是热点则限流 → 通过/拒绝
                    ↓
            统计访问频率
            (滑动窗口/计数器)
```

### 限流算法选择

| 算法 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 计数器 | 简单 | 临界问题 | 粗略限流 |
| 滑动窗口 | 精确 | 实现复杂 | 精度要求高 |
| 令牌桶 | 允许突发 | 需要额外存储 | 允许突发流量 |
| 漏桶 | 稳定输出 | 不允许突发 | 严格限流 |

## ④ Hard - 难点 (挑战)

### 可能遇到的问题

**难点1：热点识别延迟**
```
场景：商品刚上架就爆火，前10秒无法识别为热点
→ 解决：热点识别采用异步统计，启动后快速识别

难点2：热点参数动态变化
场景：昨天的热点是 iPhone，今天变成华为
→ 解决：热点列表需要动态更新，定期刷新热点

难点3：参数值过多，存储压力大
场景：用户ID有1000万个，无法全部存储
→ 解决：只存储 Top N 热点，使用 LRU 淘汰

难点4：分布式环境热点统计
场景：3台服务器，每台统计自己的热点
→ 解决：使用 Redis 集中统计，或 Sentinel 集群限流

难点5：热点与普通请求冲突
场景：热点商品限流，影響普通商品
→ 解决：参数级限流，只影响热点参数值
```

### 边界情况

| 边界情况 | 处理方式 |
|----------|----------|
| 参数值为空 | 跳过，不进行限流 |
| 参数值超长 | 截断或哈希 |
| 热点 key 过期 | 重新开始统计 |
| Redis 不可用 | 降级为本地限流 / 直接放行 |

### 候选方案对比

| 方案 | 优点 | 缺点 | 适用场景 |

## ⑦ Impl - 实现 (细节)

### yudao-cloud 现有实现分析

**现状**：yudao-cloud 目前集成了 **RateLimiter**（限流）功能，位于：
- 路径：`yudao-framework/yudao-spring-boot-starter-protection/`
- 实现：基于 Redis 的分布式限流

**当前限流注解使用方式**：

```java
@RateLimiter(time = 1, count = 100, 
             keyResolver = UserRateLimiterKeyResolver.class)
public UserVO getUser(Long id) {
    // 业务逻辑
}
```

**当前不支持的功能**：
- ❌ 热点参数识别
- ❌ 参数级限流
- ❌ 热点参数熔断

### 集成 Sentinel 实现热点参数保护

#### Step 1: 添加依赖

```xml
<!-- pom.xml -->
<dependency>
    <groupId>com.alibaba.csp</groupId>
    <artifactId>sentinel-core</artifactId>
    <version>1.8.8</version>
</dependency>
<dependency>
    <groupId>com.alibaba.csp</groupId>
    <artifactId>sentinel-parameter-flow-control</artifactId>
    <version>1.8.8</version>
</dependency>
```

#### Step 2: 配置启动类

```java
@Configuration
public class SentinelConfig {

    @PostConstruct
    public void init() {
        // 初始化 ParamFlowRuleManager
        initHotParamRules();
    }

    private void initHotParamRules() {
        // 为 getProduct 接口配置热点参数规则
        ParamFlowRule rule = new ParamFlowRule("getProduct")
            .setParamIdx(0)  // 第一个参数是 productId
            .setCount(100)   // 每个商品ID每秒最多100请求
            .setControlBehavior(ParamFlowRule.CONTROL_BEHAVIOR_RATE_LIMITER)
            .setMaxQueueingTimeMs(500);

        ParamFlowRuleManager.loadRules(Collections.singletonList(rule));
    }
}
```

#### Step 3: 定义资源

```java
@Service
public class ProductService {

    @SentinelResource(value = "getProduct", 
                      blockHandler = "getProductBlocked")
    public ProductVO getProduct(Long productId) {
        // 查询商品
        return productMapper.selectById(productId);
    }

    // 限流降级处理
    public ProductVO getProductBlocked(Long productId, BlockException e) {
        // 返回默认结果或抛出异常
        throw new ServiceException(429, "请求过于频繁，请稍后重试");
    }
}
```

#### Step 4: 配置控制台（可选）

```yaml
# application.yml
spring:
  cloud:
    sentinel:
      transport:
        dashboard: localhost:8080  # Sentinel 控制台地址
      eager: true  # 启动时加载规则
```

### 验证步骤

```
Step 1: 检查限流注解生效
  → 校验点：@RateLimiter 注解被正确识别
  → 验证方法：查看日志，尝试触发限流

Step 2: 测试热点参数限流
  → 校验点：对同一商品ID请求超过阈值被限流
  → 验证方法：使用 JMeter 模拟1000次请求

Step 3: 验证降级逻辑
  → 校验点：触发限流时返回降级响应
  → 验证方法：检查返回的 HTTP 状态码和响应体

Step 4: 检查监控数据
  → 校验点：Sentinel 控制台显示实时数据
  → 验证方法：登录控制台查看
```

## ⑧ SKILL - 提炼 (复用)

### 触发条件

什么场景下用这个 SKILL？

```
✅ 场景1：热点商品查询（如双11秒杀）
✅ 场景2：热门用户动态（如大V微博）
✅ 场景3：高频接口保护（如搜索接口）
✅ 场景4：API 防刷（如短信验证码）
```

### 执行流程

```
Step 1: 评估是否需要热点保护
  - 是否有热点数据的可能？
  - 访问频率是否可能超过系统承受能力？

Step 2: 确定保护参数
  - 哪个参数是热点？
  - 参数索引是多少？

Step 3: 配置限流规则
  - 设置阈值（建议 100-500 QPS）
  - 设置降级策略

Step 4: 编写业务代码
  - 添加 @SentinelResource 注解
  - 实现 blockHandler 降级逻辑

Step 5: 测试验证
  - 模拟高并发测试
  - 观察限流效果
```
