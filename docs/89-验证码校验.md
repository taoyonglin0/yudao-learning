## ① Why - 价值 (为什么)

### 背景与痛点

在用户注册、登录、密码重置等敏感操作中，如何确认操作者确实是手机号持有人？

**不用验证码校验的痛点：**

| 痛点 | 说明 |
|------|------|
| **账户被盗风险** | 黑客可以通过暴力猜解或社工库获取他人账号密码 |
| **批量注册** | 恶意用户可以批量注册虚假账号 |
| **无法确认身份** | 无法确定操作者是手机号真正的所有者 |
| **暴力破解** | 攻击者可以无限尝试密码 |

### 验证码校验带来的价值

| 价值 | 说明 |
|------|------|
| **身份确认** | 确保操作者是手机号持有人 |
| **安全增强** | 弥补密码泄露导致的风险 |
| **防批量操作** | 限制单个手机号的操作频率 |
| **可追溯** | 验证码使用记录可查 |

## ② What - 定义 (是什么)

### 一句话定义

**验证码校验** 是验证用户输入的短信验证码是否正确、是否过期、是否已使用的过程，确保只有真正的手机号持有人才能完成敏感操作。

### 核心概念

| 概念 | 说明 |
|------|------|
| **SmsCode** | 短信验证码实体，存储验证码及状态 |
| **验证码状态** | 未使用(used=false)、已使用(used=true)、已过期 |
| **验证码有效期** | 验证码创建后允许使用的时间窗口 |
| **场景(Scene)** | 验证码使用场景（登录、注册、找回密码等） |

### 核心代码结构

```
SmsCodeService
  ├── sendSmsCode()     - 发送验证码（创建+发送）
  ├── useSmsCode()     - 使用验证码（校验+标记已使用）
  └── validateSmsCode() - 校验验证码（仅校验，不标记）
```

## ③ How - 思维 (怎么做)

### 数据模型设计

#### 短信验证码表（system_sms_code）

```sql
CREATE TABLE `system_sms_code` (
  `id` bigint NOT NULL AUTO_INCREMENT COMMENT 'ID',
  `mobile` varchar(20) NOT NULL COMMENT '手机号',
  `code` varchar(20) NOT NULL COMMENT '验证码',
  `scene` int NOT NULL COMMENT '场景',
  `today_index` int NOT NULL COMMENT '当天发送序号',
  `create_ip` varchar(50) NOT NULL COMMENT '创建IP',
  `used` bit(1) NOT NULL DEFAULT b'0' COMMENT '是否使用',
  `used_time` datetime DEFAULT NULL COMMENT '使用时间',
  `used_ip` varchar(50) DEFAULT NULL COMMENT '使用IP',
  `create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  PRIMARY KEY (`id`),
  KEY `idx_mobile_scene` (`mobile`,`scene`),
  KEY `idx_mobile_code` (`mobile`,`code`)
) ENGINE=InnoDB COMMENT='短信验证码';
```

#### 核心字段说明

| 字段 | 说明 |
|------|------|
| mobile | 手机号 |
| code | 验证码值（明文或加密存储） |
| scene | 场景枚举（1-登录，2-注册，3-找回密码等） |
| today_index | 当天发送序号，用于计算每日发送上限 |
| used | 是否已使用 |
| create_time | 创建时间，用于判断是否过期 |

### 关键流程设计

#### 验证码校验流程

```
Step 1: 查询验证码记录
  输入: mobile, code, scene
  SQL: select * from system_sms_code 
       where mobile=? and code=? and scene=?
       order by create_time desc limit 1
  
Step 2: 校验验证码是否存在
  判断: lastSmsCode == null
  结果: 不存在 -> 抛出 SMS_CODE_NOT_FOUND

Step 3: 校验是否过期
  判断: now - createTime > expireTime
  结果: 已过期 -> 抛出 SMS_CODE_EXPIRED

Step 4: 校验是否已使用
 判断: used == true
  结果: 已使用 -> 抛出 SMS_CODE_USED

Step 5: 返回校验结果
  返回: SmsCodeDO 实体，后续业务使用
```

#### 验证码使用流程（useSmsCode）

```
Step 1: 调用 validateSmsCode0 进行校验
  → 同上Step 1-4的校验逻辑

Step 2: 标记为已使用
  SQL: update system_sms_code 
       set used=true, used_time=now, used_ip=? 
       where id=?

Step 3: 返回结果
  → 校验通过，业务继续执行
```

### 配置属性

@Data

## ④ Hard - 难点 (挑战)

### 难点一：时序问题

**场景**：用户点击发送验证码后，立即点击验证

**问题**：验证码刚创建，还未写入数据库，导致校验失败

**解决**：
```java
// 验证码创建和插入数据库在同一事务中
// 读取时使用最新的记录（order by create_time desc limit 1）
```

### 难点二：验证码暴力破解

**场景**：攻击者遍历所有6位数组合尝试破解

**解决**：
1. **频率限制**：同一手机号每分钟只能发送一次
2. **IP限制**：同一IP每天发送数量有限制
3. **错误次数**：验证码错误多次后进行锁定

### 难点三：验证码重放攻击

**场景**：攻击者截获已使用的验证码再次使用

**解决**：
1. **一次性使用**：验证码使用后立即标记为 used=true
2. **有效期控制**：验证码设置较短的有效期（如5分钟）
3. **使用记录**：记录 used_ip 便于追溯

### 难点四：时区问题

**场景**：服务器时区与用户时区不一致

**问题**：today_index 计算错误，导致每日限制失效

**解决**：
```java
// 使用 isToday() 方法判断是否是今天
// 该方法基于服务器时区计算
LocalDateTime now = LocalDateTime.now();
boolean isToday = lastSmsCode.getCreateTime().toLocalDate().equals(now.toLocalDate());
```

## ⑥ Select - 选型 (选哪个)

### 候选方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **内存验证码** | 无需存储、速度快 | 无法限频、易泄露 | 开发测试 |
| **数据库存储** | 可查询、可统计 | 性能一般 | 中小项目 |
| **Redis存储** | 高性能、可设置过期 | 需要额外组件 | 大型项目 |
| **第三方校验** | 安全、稳定 | 成本高、依赖外部 | 高安全场景 |

### 选型理由

**选择 yudao-cloud 的数据库方案，因为：**

1. **实现完整**：自带发送频率、每日上限、防重放等完整功能
2. **可追溯**：验证码记录可查，便于问题排查
3. **易集成**：Spring Boot 配置方式，开箱即用
4. **可扩展**：可方便增加 IP 限频等功能

## ⑦ Impl - 实现 (细节)

### yudao-cloud 项目中的实现

#### 核心实现类

```java
@Service
@Validated
public class SmsCodeServiceImpl implements SmsCodeService {

    @Resource
    private SmsCodeProperties smsCodeProperties;

    @Resource
    private SmsCodeMapper smsCodeMapper;

    @Resource
    private SmsSendService smsSendService;

    @Override
    public void sendSmsCode(SmsCodeSendReqDTO reqDTO) {
        // 1. 获取场景枚举
        SmsSceneEnum sceneEnum = SmsSceneEnum.getCodeByScene(reqDTO.getScene());
        
        // 2. 校验发送频率和每日上限
        SmsCodeDO lastSmsCode = smsCodeMapper.selectLastByMobile(mobile, null, null);
        if (lastSmsCode != null) {
            // 检查发送频率
            if (between(lastSmsCode.getCreateTime(), now) < sendFrequency) {
                throw exception(SMS_CODE_SEND_TOO_FAST);
            }
            // 检查每日上限
            if (isToday(lastSmsCode.getCreateTime()) && 
                lastSmsCode.getTodayIndex() >= maxPerDay) {
                throw exception(SMS_CODE_EXCEED_SEND_MAXIMUM_QUANTITY_PER_DAY);
            }
        }
        
        // 3. 创建验证码
        String code = generateCode();
        
        // 4. 发送短信
        smsSendService.sendSingleSms(mobile, templateCode, params);
    }

    @Override
    public void useSmsCode(SmsCodeUseReqDTO reqDTO) {
        // 1. 校验验证码
        SmsCodeDO lastSmsCode = validateSmsCode0(reqDTO.getMobile(), reqDTO.getCode(), reqDTO.getScene());
        
        // 2. 标记已使用
        smsCodeMapper.updateById(SmsCodeDO.builder()
            .id(lastSmsCode.getId())
            .used(true)
            .usedTime(LocalDateTime.now())
            .usedIp(reqDTO.getUsedIp())
            .build());
    }

    private SmsCodeDO validateSmsCode0(String mobile, String code, Integer scene) {
        // 1. 查询最新验证码
        SmsCodeDO lastSmsCode = smsCodeMapper.selectLastByMobile(mobile, code, scene);
        
        // 2. 不存在
        if (lastSmsCode == null) {
            throw exception(SMS_CODE_NOT_FOUND);
        }
        
        // 3. 已过期
        if (between(lastSmsCode.getCreateTime(), now) >= expireTimes.toMillis()) {
            throw exception(SMS_CODE_EXPIRED);
        }
        
        // 4. 已使用
        if (Boolean.TRUE.equals(lastSmsCode.getUsed())) {
            throw exception(SMS_CODE_USED);
        }
        
        return lastSmsCode;
    }
}
```

## ⑧ SKILL - 提炼 (复用)

### 触发条件

```
场景1: 用户登录时需要短信验证码
场景2: 用户注册时需要短信验证码
场景3: 找回密码时需要短信验证码
场景4: 敏感操作二次验证（如修改手机号）
```

### 执行流程

```
Step 1: 调用 sendSmsCode 发送验证码
  - 参数: mobile, scene, createIp
  - 内部校验: 发送频率、每日上限

Step 2: 用户输入验证码
  - 前端将 mobile + code + scene 发送给后端

Step 3: 调用 useSmsCode 校验并使用
  - 校验: 验证码是否存在、是否过期、是否已使用
  - 使用: 标记为已使用，记录使用时间和IP
  - 返回: 校验通过，业务继续
```
