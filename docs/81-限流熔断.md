# 81-限流熔断

> 网关 - 限流熔断核心原理与实现

---

## ① Why - 价值

**背景与痛点**：
- 原来不做限流，双11零点秒杀活动时，数据库被打挂
- 恶意用户高频请求接口，单机QPS从1000飙升到10000
- 下游服务挂了，上游服务也被拖垮，形成雪崩

**收益**：
- 保护系统不被流量冲垮
- 保障核心服务可用性
- 防止故障扩散

**用户**：
- 网关层、API层、后端服务

---

## ② What - 定义

**一句话定义**：
限流熔断是一种保护机制，限流控制请求速率，熔断在故障时快速失败防止雪崩。

**核心组成**：
1. **限流器** (Rate Limiter) - 控制请求速率
2. **熔断器** (Circuit Breaker) - 故障快速失败
3. **降级处理** (Fallback) - 异常时的后备方案

**关键术语**：
- QPS - 每秒请求数
- RRateLimiter - Redisson分布式限流器
- Key Resolver - 限流Key解析器

## ③ How - 思维

**目标**：实现分布式限流能力

**范围**：
- 允许：yudao-framework/yudao-spring-boot-starter-protection/
- 禁止：改动业务模块的核心业务逻辑

**数据模型**：

```yaml
# 限流配置示例
rate.limiter:
  enabled: true
  default:
    count: 100  # 每秒允许100次
    time: 1     # 时间窗口1秒
```

**核心流程**：
```
请求进入 
  → 解析限流Key 
  → 尝试获取令牌 
  → 获取成功→执行业务 
  → 获取失败→返回429
```

## ④ Hard - 难点

**难点1：分布式限流，多台机器如何保证一致性？**
```
场景：服务部署了3台，如何保证全局限流？
→ 使用Redis分布式限流，所有机器访问同一个Redis
```

**难点2：限流Key如何设计？**
```
场景：按用户限流？按IP限流？按接口限流？
→ 提供多种Key Resolver：Default/User/IP/ServerNode/Expression
```

**难点3：熔断器如何实现？**
```
场景：下游服务挂了，如何快速失败？
→ yudao未使用Resilience4j/Sentinel，通过Nacos配置开关实现
```

## ⑥ Select - 选型

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| Redisson RRateLimiter | 与Redis集成、性能高 | 需要Redis | 高并发场景 |
| Sentinel | 阿里开源、功能全 | 学习成本高 | 阿里技术栈 |
| Resilience4j | 轻量、Java8+ | 无分布式支持 | 单机场景 |
| Nginx限流 | 网关层限流 | 需配置Lua | 入口限流 |

**选型理由**：
1. 项目已有Redisson，集成成本低
2. 使用RRateLimiter，支持分布式限流
3. 提供多种Key Resolver，灵活配置

## ⑦ Impl - 实现

**数据模型**：

```java
// RateLimiter 注解
@RateLimiter(time = 1, count = 100, keyResolver = UserRateLimiterKeyResolver.class)
public void query() { }
```

**核心代码**：

```java
// RateLimiterAspect 核心逻辑
public void beforePointCut(JoinPoint joinPoint, RateLimiter rateLimiter) {
    // 1. 解析Key
    String key = keyResolver.resolver(joinPoint, rateLimiter);
    // 2. 尝试获取令牌
    boolean success = rateLimiterRedisDAO.tryAcquire(key,
            rateLimiter.count(), rateLimiter.time(), rateLimiter.timeUnit());
    // 3. 获取失败抛异常
    if (!success) {
        throw new ServiceException(TOO_MANY_REQUESTS);
    }
}
```

```java
// RateLimiterRedisDAO 使用Redisson限流
public Boolean tryAcquire(String key, int count, int time, TimeUnit timeUnit) {
    RRateLimiter rateLimiter = getRRateLimiter(key, count, time, timeUnit);
    return rateLimiter.tryAcquire();
}
```

**关键Key Resolver**：
- DefaultRateLimiterKeyResolver - 全局限流
- UserRateLimiterKeyResolver - 用户级限流
- ClientIpRateLimiterKeyResolver - IP级限流
- ServerNodeRateLimiterKeyResolver - 节点级限流
- ExpressionRateLimiterKeyResolver - 自定义表达式

**配置示例**：

```yaml
# application.yml
rate-limiter:
  default:
    count: 100
    time: 1
```

## ⑧ SKILL - 提炼

**触发条件**：
- 场景1：需要限制接口请求速率
- 场景2：需要防止高频攻击
- 场景3：需要保护下游服务

**执行流程**：
```
Step 1: 引入依赖
  - 已有 yudao-spring-boot-starter-protection
  - 已有 Redisson

Step 2: 使用限流注解
  - @RateLimiter(time = 1, count = 100)
  - 指定 keyResolver

Step 3: 配置限流规则
  - 通过 Nacos 配置中心动态调整
  - 无需重启服务
```

**配方**：
- 技术栈：Java 8+, Spring Boot 2.7+, Redisson
- 依赖包：yudao-spring-boot-starter-protection
- 配置：rate-limiter.* (application.yml)

**使用示例**：
```java
// 1. 用户级限流：每个用户每秒100次
@RateLimiter(time = 1, count = 100, keyResolver = UserRateLimiterKeyResolver.class)
public void query() { }

// 2. IP级限流：每个IP每秒50次
@RateLimiter(time = 1, count = 50, keyResolver = ClientIpRateLimiterKeyResolver.class)
public void submit() { }

// 3. 接口级限流：每个接口每秒1000次
@RateLimiter(time = 1, count = 1000)
public void list() { }
```

**验收标准**：
- [ ] 功能正常：限流注解能正常拦截
- [ ] 单元测试通过：核心方法测试通过
- [ ] 性能达标：限流判断<100ms
- [ ] 日志可查：拦截日志可查询
- [ ] 配置可动态调整：通过Nacos修改生效
