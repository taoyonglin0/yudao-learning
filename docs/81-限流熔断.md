## ① Why - 价值 (为什么)

### 背景与痛点

在分布式微服务架构中，系统面临的核心挑战：

1. **流量突增导致雪崩**：秒杀活动、热点事件可能带来瞬时 10-100 倍的流量激增
2. **后端服务能力有限**：数据库连接池、线程池资源有限，无法支撑无限请求
3. **级联故障传播**：某个服务不可用可能拖垮整个调用链

### 收益分析

| 收益项 | 预期效果 |
|--------|----------|
| 系统稳定性 | 99.9% 可用性保障 |
| 资源保护 | 数据库连接池不被打满 |
| 用户体验 | 限流拒绝比超时等待更好 |
| 故障隔离 | 防止级联雪崩 |

### 使用场景

- API 接口限流（如登录、查询）
- 定时任务并发控制
- 第三方调用频率限制
- 秒杀/抢购场景

## ② What - 定义 (是什么)

### 核心概念

**限流 (Rate Limiting)**：控制单位时间内的请求数量，防止系统过载

**熔断 (Circuit Breaker)**：当下游服务故障时，快速失败而不是等待超时（yudao-cloud 当前版本主要关注限流，熔断能力可结合 Sentinel 实现）

### 一句话定义

> 限流是通过 Redisson 分布式限流器，在方法级别控制接口调用频率的保护机制。

### 核心组成部分

```
┌─────────────────────────────────────────┐
│           限流保护体系                    │
├─────────────────────────────────────────┤
│  @RateLimiter 注解                      │
│       ↓                                 │
│  RateLimiterAspect (AOP 拦截器)          │
│       ↓                                 │
│  RateLimiterKeyResolver (Key 解析器)    │
│       ↓                                 │
│  RateLimiterRedisDAO (Redis 限流实现)     │
└─────────────────────────────────────────┘
```

### 关键术语

| 术语 | 说明 |
|------|------|
| RRateLimiter | Redisson 分布式限流器 |
| RateType.OVERALL | 全局限流类型 |
| keyResolver | 限流 key 解析策略 |

## ③ How - 思维 (怎么做)

### 数据模型设计

限流不依赖持久化表结构，通过 Redis 动态管理：

```
Redis Key 格式：rate_limiter:{md5_key}
Redis Type：RRateLimiter
过期时间：time 参数指定的时间窗口
```

### 关键流程设计

```
请求进来
    ↓
1. 解析 @RateLimiter 注解参数
    ↓
2. 根据 keyResolver 获取限流 Key
    ↓
3. 调用 Redisson RRateLimiter.tryAcquire()
    ↓
4. 获取令牌成功 → 放行
    ↓
5. 获取失败 → 抛出 TOO_MANY_REQUESTS 异常
```

### 关键代码设计

#### 1. 注解定义

```java
@Target({ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
public @interface RateLimiter {
    int time() default 1;           // 时间窗口
    TimeUnit timeUnit() default TimeUnit.SECONDS;
    int count() default 100;      // 限流次数
    String message() default "";     // 提示信息
    Class<? extends RateLimiterKeyResolver> keyResolver() default DefaultRateLimiterKeyResolver.class;
    String keyArg() default "";    // Key 参数
}
```

#### 2. AOP 拦截实现

```java
@Aspect
public class RateLimiterAspect {
    
    @Before("@annotation(rateLimiter)")
    public void beforePointCut(JoinPoint joinPoint, RateLimiter rateLimiter) {
        // 1. 获取 Key 解析器
        RateLimiterKeyResolver keyResolver = keyResolvers.get(rateLimiter.keyResolver());
        // 2. 解析限流 Key
        String key = keyResolver.resolver(joinPoint, rateLimiter);
        // 3. 尝试获取令牌
        boolean success = rateLimiterRedisDAO.tryAcquire(key,
                rateLimiter.count(), rateLimiter.time(), rateLimiter.timeUnit());
        // 4. 获取失败则抛异常
        if (!success) {
            throw new ServiceException(TOO_MANY_REQUESTS);
        }
    }
}
```

#### 3. Redis 限流实现

```java
public class RateLimiterRedisDAO {
    
    public Boolean tryAcquire(String key, int count, int time, TimeUnit timeUnit) {
        RRateLimiter rateLimiter = getRRateLimiter(key, count, time, timeUnit);
        return rateLimiter.tryAcquire();  // 获取 1 个令牌
    }
    
    private RRateLimiter getRRateLimiter(String key, long count, int time, TimeUnit timeUnit) {
        String redisKey = formatKey(key);
        RRateLimiter rateLimiter = redissonClient.getRateLimiter(redisKey);
        
        // 首次设置限流规则
        if (rateLimiter.getConfig() == null) {
            rateLimiter.trySetRate(RateType.OVERALL, count, Duration.ofSeconds(time));
        }
        return rateLimiter;
    }
}
```

#### 4. Key 解析策略（5 种）

| 解析器 | 适用场景 | Key 组成 |
|--------|----------|----------|
| DefaultRateLimiterKeyResolver | 全局限流 | 方法签名 + 参数 MD5 |
| UserRateLimiterKeyResolver | 登录用户 | userId |
| ClientIpRateLimiterKeyResolver | IP 级别 | 请求 IP |
| ServerNodeRateLimiterKeyResolver | 服务节点 | 节点 ID |
| ExpressionRateLimiterKeyResolver | 自定义 | SPEL 表达式 |

## ④ Hard - 难点 (挑战)

### 挑战 1：分布式环境下的限流一致性

**场景**：多节点部署时，每个节点都计算自己的限流？

**解决方案**：使用 Redisson 分布式限流器，所有节点共享同一个 Redis 中的限流状态

**难点**：首次创建限流器时的竞态条件

```java
// 双重检查锁定
if (config == null) {
    rateLimiter.trySetRate(...);  // 竞态点
    rateLimiter.expire(duration); // 确保过期
}
```

### 挑战 2：限流 Key 的设计

**场景**：同一个接口，不同用户应该独立限流？

**解决方案**：选择合适的 KeyResolver

```java
// 用户级别限流
@RateLimiter(time = 60, count = 10, 
           keyResolver = UserRateLimiterKeyResolver.class)
public void sendSms() { }
```

### 挑战 3：限流阈值配置

**场景**：想改限流阈值不想重启服务？

**当前方案**：需要通过配置中心动态配置（代码层面未实现热更新）

**替代方案**：使用 expression keyArg 动态计算

## ⑥ Select - 选型 (选哪个)

### 候选方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| Redisson RRateLimiter | 分布式、文档全 | 依赖 Redis | 微服务限流 |
| Bucket4j | 无需额外部件 | 仅单机 | 单机限流 |
| Sentinel | 限流+熔断+降级 | 学习曲线��� | 流量保护 |
| Guava RateLimiter | 简单易用 | 仅单机 | 简单场景 |
| Nginx | 网关层限流 | 运维成本高 | 基础设施限流 |

### 选型理由

**yudao-cloud 选择 Redisson RRateLimiter**：

1. **分布式支持**：基于 Redis 实现多节点共享状态
2. **无额外组件**：项目已引入 Redisson（用于分布式锁）
3. **配置简单**：通过注解即可使用
4. **性能优秀**：Redis Lua 脚本执行，延迟 < 1ms
5. **生态集成**：与 Spring Boot 深度集成

```xml
<!-- 已有依赖 -->
<dependency>
    <groupId>org.redisson</groupId>
    <artifactId>redisson-spring-boot-starter</artifactId>
</dependency>
```

### 失败恢复机制

| 场景 | 处理方式 |

| 限流器未初始化 | 首次自动创建 |
| 限流 Key 冲突 | 使用 MD5 压缩 |

## ⑧ SKILL - 提炼 (复用)

### 触发条件

什么场景下使用这个技能？

```
场景1：API 接口需要限流
场景2：防止定时任务并发执行
场景3：第三方调用频率限制
```

### 执行流程

```
Step 1: 确认已引入依赖
  redisson-spring-boot-starter

Step 2: 在方法上添加注解
  @RateLimiter(time = 1, count = 100)

Step 3: 选择 Key 策略
  - 默认：全部用户共享
  - 用户：每个用户独立
  - IP：每个 IP 独立

Step 4: 验证
  - 正常请求通过
  - 超限请求被拒
```

### 技术配方

- **依赖**：`redisson-spring-boot-starter`
- **环境**：Redis + Spring Boot
- **注解**：`@RateLimiter`

### 使用配方

```java
// 配方 1：基础限流
@RateLimiter(time = 1, count = 100)
public void api() { }

// 配方 2：用户限流
@RateLimiter(time = 60, count = 10, 
           keyResolver = UserRateLimiterKeyResolver.class)
public void sensitiveApi() { }

// 配方 3：IP 限流
@RateLimiter(time = 60, count = 100,
           keyResolver = ClientIpRateLimiterKeyResolver.class)
public void publicApi() { }
```

### 注意事项

1. **限流粒度**：方法级别，适合 API 接口
2. **Key 唯一性**：确保 Key 不会冲突
3. **阈值设置**：根据实际业务调整
4. **Redis 依赖**：限流器依赖 Redis 可用性

## 附录：进阶 - 熔断实现

当前 yudao-cloud 使用限流来保护系统，熔断能力可以通过以下方式扩展：

### Sentinel 整合

```java
// 引入 Sentinel
<dependency>
    <groupId>com.alibaba.csp</groupId>
    <artifactId>sentinel-core</artifactId>
</dependency>

// 定义资源
@SentinelResource(value = "api", blockHandler = "blockHandler")
public Result api() { }

public Result blockHandler(BlockException e) {
    return Result.error("服务繁忙，请稍后重试");
}
```

### 熔断策略

| 策略 | 说明 |
|------|------|
| 慢调用比例 | 响应时间 > 阈值比例 > 比例 |
| 错误比例 | 异常比例 > 比例 |
| 错误数 | 异常数 > 数量 |
