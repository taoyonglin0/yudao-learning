## ① Why - 价值 (为什么)

### 背景与痛点

在微服务架构下，一个业务操作往往涉及多个服务/数据库：

```
用户下单场景：
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   订单服务   │───▶│   库存服务   │───▶│   支付服务   │
│  (订单库)   │    │  (库存库)   │    │  (支付库)   │
└─────────────┘    └─────────────┘    └─────────────┘
```

**不用分布式事务的痛点：**

1. **数据不一致**：订单创建成功，库存扣减失败，但订单已落库 → 库存少了但钱没付
2. **人工修复成本高**：需要人工对账、补录、补偿，耗时耗力
3. **用户体验差**：用户付款成功但订单显示失败，或者反过来
4. **业务风险**：资金类业务如果出现数据不一致，可能造成资损

### AT模式带来的收益

| 收益 | 说明 |
|------|------|
| **数据一致性** | 保证多个服务要么全部成功，要么全部回滚 |
| **开发无感知** | 业务代码无需关心分布式事务细节，零侵入 |
| **性能较高** | 一阶段直接提交本地事务，二阶段异步化 |
| **自动回滚** | 失败时自动根据Undo Log回滚，无需手动补偿 |

### 适用场景

- 跨库操作：同一个服务访问多个数据库
- 跨服务操作：订单服务调用库存服务、支付服务
- 混合场景：跨服务 + 跨库

## ② What - 定义 (是什么)

### 一句话定义

**AT 模式（Automatic Transaction）** 是 Seata 分布式事务解决方案中的一种模式，基于支持本地 ACID 事务的关系型数据库，通过 undo log（回滚日志）实现自动回滚的两阶段提交协议。

### 核心概念

| 概念 | 说明 |
|------|------|
| **TC (Transaction Coordinator)** | 事务协调器，负责管理全局事务状态 |
| **TM (Transaction Manager)** | 事务管理器，发起全局事务，控制边界 |
| **RM (Resource Manager)** | 资源管理器，管理分支事务，与数据库交互 |
| **XID (Global Transaction ID)** | 全局事务ID，串联所有分支事务 |
| **Branch ID** | 分支事务ID |
| **Undo Log** | 回滚日志，记录前后镜像用于回滚 |

### 核心组成

```
┌─────────────────────────────────────────────────────┐
│                    Seata Server (TC)                 │
│              事务协调中心 (独立部署)                   │
└─────────────────────┬───────────────────────────────┘
                      │
        ┌─────────────┴─────────────┐
        │                           │
        ▼                           ▼
┌───────────────────┐     ┌───────────────────┐
│   服务A (TM+RM)   │     │   服务B (TM+RM)   │
│  ─────────────    │     │  ─────────────    │
│  订单数据库        │     │  库存数据库        │
│  (RM)             │     │  (RM)             │
└───────────────────┘     └───────────────────┘
```

## ③ How - 思维 (怎么做)

### 数据模型设计

#### Undo Log 表（必须）

每个参与分布式事务的数据库都需要创建 Undo Log 表：

```sql
CREATE TABLE `undo_log` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT,
  `branch_id` bigint(20) NOT NULL,
  `xid` varchar(100) NOT NULL,
  `context` varchar(128) NOT NULL,
  `rollback_info` longblob NOT NULL,
  `log_status` int(11) NOT NULL,
  `log_created` datetime NOT NULL,
  `log_modified` datetime NOT NULL,
  PRIMARY KEY (`id`),
  UNIQUE KEY `ux_undo_log` (`xid`, `branch_id`)
) ENGINE=InnoDB AUTO_INCREMENT=1 DEFAULT CHARSET=utf8;
```

**字段说明：**

| 字段 | 说明 |
|------|------|
| xid | 全局事务ID，格式：`ip:port:timestamp:sequence` |
| branch_id | 分支事务ID |
| rollback_info | 回滚信息（JSON，包含前后镜像） |
| log_status | 日志状态：0-正常，1-已删除 |

#### 回滚日志结构示例

```json
{
  "branchId": 641789253,
  "xid": "192.168.1.100:8091:123456789",
  "undoItems": [
    {
      "tableName": "product",
      "sqlType": "UPDATE",
      "beforeImage": {
        "rows": [{
          "fields": [
            {"name": "id", "type": 4, "value": 1},
            {"name": "name", "type": 12, "value": "TXC"},
            {"name": "since", "type": 12, "value": "2014"}
          ]
        }]
      },
      "afterImage": {
        "rows": [{
          "fields": [
            {"name": "id", "type": 4, "value": 1},
            {"name": "name", "type": 12, "value": "GTS"},
            {"name": "since", "type": 12, "value": "2014"}
          ]
        }]
      }
    }
  ]
}
```

### 关键流程设计

#### 整体流程图

```
┌────────────────────────────────────────────────────────────────────┐
│                         全局事务生命周期                             │
└────────────────────────────────────────────────────────────────────┘

  TM (业务发起)                    TC (协调者)                      RM (各服务)
      │                               │                               │
      │───── 开启全局事务 (XID) ──────▶│                               │
      │                               │                               │
      │                               │──── 注册分支 ────────────────▶│
      │                               │                               │
      │                               │◀──── 成功响应 ────────────────│
      │                               │                               │
      │──── 执行业务 ────────────────▶│                               │
      │                               │                               │
      │                               │──── 一阶段提交 ──────────────▶│
      │                               │    (本地事务+UndoLog)         │
      │                               │                               │
      │                               │◀──── 提交成功 ────────────────│
      │                               │                               │
      │──── 业务完成                  │                               │
      │                               │                               │
      │                               │──── 二阶段提交/回滚 ─────────▶│
      │                               │    (异步删除/重做)            │
      │                               │                               │
      ◀──── 最终结果 ────────────────│                               │
                                    │                               │
```

#### 一阶段：执行与记录

```
Step 1: 解析SQL
  输入: update product set name = 'GTS' where name = 'TXC'
  输出: SQL类型=UPDATE, 表=product, 条件=name='TXC'

Step 2: 查询前镜像 (Before Image)
  SQL: select id, name, since from product where name = 'TXC'
  结果: {id:1, name:'TXC', since:'2014'}

Step 3: 执行业务SQL
  SQL: update product set name = 'GTS' where name = 'TXC'
  结果: 影响行数=1

Step 4: 查询后镜像 (After Image)
  SQL: select id, name, since from product where id = 1
  结果: {id:1, name:'GTS', since:'2014'}

Step 5: 生成回滚日志
  记录: 前镜像 + 后镜像 + SQL信息 -> undo_log表

Step 6: 注册分支 (获取全局锁)
  向TC申请: product表主键=1的记录的全局锁
  目的: 防止脏写

Step 7: 提交本地事务
  提交: 业务数据 + undo_log 一起提交
  释放: 本地锁和连接资源

Step 8: 上报结果
  报告: 分支事务执行结果给TC
```

#### 二阶段：提交

```
二阶段-提交流程:
  1. TC收到分支提交请求
  2. 放入异步队列 (快速响应)
  3. 异步批量删除undo_log
  4. 释放全局锁
```

#### 二阶段：回滚

```
二阶段-回滚流程:
  1. TC发送分支回滚请求
  2. RM开启本地事务
  3. 根据XID查询undo_log
  4. 数据校验 (后镜像 vs 当前数据)
  5. 生成反向SQL: update product set name='TXC' where id=1
  6. 执行反向SQL
  7. 提交本地事务
  8. 删除undo_log
  9. 上报结果给TC
```

## ④ Hard - 难点 (挑战)

### 难点一：写隔离问题

**场景**：两个全局事务同时操作同一行数据

```
时间线:
T1: tx1 开启本地事务，更新 m=900，尝试获取全局锁 (product.id=1)
T2: tx2 开启本地事务，更新 m=800，尝试获取全局锁 (product.id=1)
    → tx2 被阻塞，等待 tx1 释放全局锁
T3: tx1 本地提交，释放全局锁
T4: tx2 获取全局锁，本地提交

结果: 串行执行，无脏写
```

**如果 tx1 回滚**：
```
T1: tx1 开启本地事务，更新 m=900
T2: tx2 开启本地事务，更新 m=800，尝试获取全局锁 → 被阻塞
T3: tx1 本地提交前尝试获取全局锁 → 成功
T4: tx1 本地提交，释放全局锁
T5: tx2 获取全局锁，本地提交
T6: tx1 二阶段回滚，重新获取本地锁，执行反向更新 m=1000
    → tx2 仍在等待全局锁，tx1 回滚失败
T7: tx2 全局锁等待超时，回滚本地事务，释放本地锁
T8: tx1 回滚成功
```

### 难点二：读隔离问题

**问题**：AT模式默认全局隔离级别是「读未提交」

**解决**：使用 `SELECT FOR UPDATE` 语句

```java
// 会申请全局锁，其他事务无法修改
@GlobalTransactional
public void getProductForUpdate(Long productId) {
    // SELECT FOR UPDATE 会申请全局锁
    ProductDO product = productMapper.selectByIdForUpdate(productId);
    // 只有当前事务提交后，其他事务才能修改
}
```

### 难点三：回滚日志过大的问题

**场景**：批量更新/删除大量数据

**影响**：undo_log 表数据量巨大，占用存储空间

**解决**：
1. 定期清理历史 undo_log（保留7天）
2. 业务高峰期避免大批量操作
3. 考虑使用 TCC 模式替代

### 难点四：服务崩溃导致事务悬挂

**场景**：一阶段执行成功但TM未收到响应

```
问题:
  1. RM 一阶段提交成功，释放了本地锁
  2. TM 崩溃，未收到成功响应
  3. TC 认为事务失败，发送回滚指令
  4. 但数据已经被提交，无法回滚
```

**解决**：Seata 通过 TC 定时检查未完成的全局事务，主动推进或回滚

## ⑥ Select - 选型 (选哪个)

### 候选方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **Seata AT** | 无侵入、自动回滚、性能较好 | 需要undo_log表、对数据库有要求 | 大多数场景（推荐） |
| **Seata TCC** | 性能最高、无undo_log | 业务代码侵入大、需要定义三个方法 | 高性能需求、弱一致性场景 |
| **Seata Saga** | 长流程支持、无锁 | 不保证隔离性、补偿逻辑复杂 | 流程长、多服务编排 |
| **XA 两阶段** | 标准协议、保证一致性 | 性能差、长期锁资源 | 对一致性要求极高的金融场景 |

### 选型理由

**选择 Seata AT 模式，因为：**

1. **无侵入**：业务代码只需加一个注解，无需改造
2. **自动回滚**：失败时自动根据 undo_log 回滚，开发成本低
3. **社区活跃**：Seata 是 Apache 顶级项目，社区活跃、文档完善
4. **集成简单**：与 Spring Cloud、Nacos 等组件天然集成
5. **性能可接受**：一阶段本地提交，二阶段异步化，整体性能较好

## ⑦ Impl - 实现 (细节)

### yudao-cloud 项目中的实现

**注意**：经分析，当前 yudao-cloud 项目**尚未集成 Seata**，项目使用的是本地事务 + 编程式事务管理。

#### 现有事务实现方式

```java
// 使用 Spring @Transactional 注解
@Transactional(rollbackFor = Exception.class)
public void notifyOrder(PayChannelDO channel, PayOrderRespDTO notify) {
    // 业务逻辑
```

#### 分布式事务缺失的影响

当前项目架构：
- 单库操作：通过 @Transactional 保证
- 跨库操作：**可能存在数据不一致风险**

**建议**：如果业务涉及跨库/跨服务操作，应考虑引入 Seata AT 模式。



### 脚本

#### 一键部署 Seata Server

```bash
#!/bin/bash
# 部署 Seata Server 脚本

SEATA_VERSION="1.7.1"
SEATA_PORT="8091"

# 下载
wget -O /tmp/seata-server.tar.gz \
  https://github.com/seata/seata/releases/download/v${SEATA_VERSION}/seata-server-${SEATA_VERSION}.tar.gz

# 解压
mkdir -p /opt/seata
tar -xzf /tmp/seata-server.tar.gz -C /opt/seata

# 启动
nohup sh /opt/seata/bin/seata-server.sh -h 127.0.0.1 -p ${SEATA_PORT} &

echo "Seata Server started on port ${SEATA_PORT}"
```

#### 检查 Undo Log 表

```sql
-- 查看未清理的 undo_log
SELECT xid, branch_id, log_status, log_created 
FROM undo_log 
WHERE log_status = 0 
ORDER BY log_created DESC 
LIMIT 20;

-- 清理历史日志 (超过7天)
DELETE FROM undo_log 
WHERE log_created < DATE_SUB(NOW(), INTERVAL 7 DAY);
```
