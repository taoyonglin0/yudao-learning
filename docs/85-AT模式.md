# 85 - AT 模式（分布式事务）

> 基于八步法分析 AT 模式在分布式事务中的应用

---

## ① Why - 价值（为什么）

### 背景与痛点

在微服务架构下，一个业务操作往往需要跨多个服务/数据库完成。例如：用户下单时，需要同时操作：
- 订单服务（订单库）
- 库存服务（库存库）
- 支付服务（支付库）
- 积分服务（积分库）

**问题**：如果其中一个服务失败，如何保证数据一致性？

**传统方案痛点**：
- 手动编写补偿代码，工作量大易出错
- 各服务补偿逻辑不一致
- 难以追踪和调试

### AT 模式价值

AT（Auto Transaction）模式是 Seata 框架的核心模式，**自动生成补偿事务**，实现：
- **自动回滚**：提交失败时自动回滚前置操作
- **自动补偿**：无需编写补偿代码
- **强一致性**：保证跨服务数据一致性
- **一致性可见**：提供事务日志可追踪

### 适用场景

- 银行转账/支付
- 库存扣减
- 订单创建
- 跨服务数据同步

## ② What - 定义（是什么）

### AT 模式定义

**一句话**：AT 模式是 Seata 提供的**自动补偿型**分布式事务解决方案，通过**解析 SQL 识别数据修改**，自动生成逆向 SQL 实现事务回滚。

### 核心组成

```
┌─────────────────────────────────────────────────┐
│                   TC (Transaction Coordinator)   │
│                   Seata Server                 │
│  ┌─────────────────────────────────────┐    │
│  │  全局事务管理器                       │    │
│  │  - 全局事务ID管理                    │    │
│  │  - 分支事务协调                      │    │
│  │  - 回滚日志管理                     │    │
│  └─────────────────────────────────────┘    │
└─────────────────────────────────────────────────┘
        ↑↓                    ↑↓
        │                    │
┌──────┴──────┐      ┌──────┴──────┐
│  TM (Transaction Manager)  │      │  RM (Resource Manager)  │
│  业务服务                  │      │  数据库                  │
│  - 发起全局事务           │      │  - 注册分支事务          │
│  - 执行业务逻辑          │      │  - 执行SQL              │
└───────────────────────────┘      └───────────────────────────┘
```

### 关键术语

| 术语 | 说明 |
|------|------|
| TC | Transaction Coordinator，事务协调器（Seata Server） |
| TM | Transaction Manager，事务管理器（发起方业务服务） |
| RM | Resource Manager，资源管理器（数据库） |
| XID | 全局事务唯一ID，用于关联各分支事务 |
| Undo Log | 回滚日志，存储数据前后镜像 |

## ③ How - 思维（怎么做）

### 数据模型设计

#### 回滚日志表（Seata Server）

```sql
-- 全局事务表
CREATE TABLE `undo_log` (
  `id` BIGINT NOT NULL AUTO_INCREMENT,
  `gmt_create` DATETIME NOT NULL,
  `gmt_modified` DATETIME NOT NULL,
  `xid` VARCHAR(128) NOT NULL,      -- 全局事务ID
  `branch_id` BIGINT NOT NULL,      -- 分支事务ID
  `context` VARCHAR(256) NOT NULL,   -- 上下���（序列化）
  `rollback_info` LONGTEXT NOT NULL, -- 回滚信息（JSON）
  `status` TINYINT NOT NULL,        -- 状态：1正常 2已回滚
  PRIMARY KEY (`id`),
  KEY `idx_xid` (`xid`)

```

#### 分支事务表

```sql
-- 分支事务记录（业务库）
CREATE TABLE `branch_table` (
  `branch_id` BIGINT NOT NULL,
  `xid` VARCHAR(128) NOT NULL,
  `status` TINYINT NOT NULL,
  PRIMARY KEY (`branch_id`)
);
```

### 关键流程设计

```
┌──────────────────────────────────────────────────────────────┐
│                      AT 模式执行流程                         │
├──────────────────────────────────────────────────────────────┤
│  1. 开启全局事务                                             │
│     TM → TC: 开启全局事务，获取 XID                          │
│                                                              │
│  2. 执行业务 SQL                                             │
│     RM: INSERT/UPDATE/DELETE → 记录前后镜像到 Undo Log      │
│                                                              │
│  3. 提交全局事务                                            │
│     TM → TC: 提交事务                                       │
│     TC → RM: 清理回滚日志                                   │
│                                                              │
│  4. 失败回滚（任一步骤失败）                                 │
│     TC: 触发回滚                                            │
│     RM: 读取 Undo Log → 生成逆向 SQL → 执行回滚             │
└──────────────────────────────────────────────────────────────┘
```

## ④ Hard - 难点（挑战）

### 挑战1：脏数据覆盖

**场景**：分支事务A执行中，事务B同时修改同一条记录
```
时间线：
T1: 事务A: 读取 age=20 (before image)
T2: 事务B: 修改 age=30
T3: 事务A: 修改 age=25 → 记录 before:20, after:25
T4: 事务A: 回滚 → 用 before:20 覆盖 → age=20 ❌
    实际应该回滚到 T2 之前 → age=30 ❌
```

**解决方案**：使用 **多版本并发控制（MVCC）**
- 记录行锁（SELECT FOR UPDATE）
- 回滚时校验 after_image 是否一致

### 挑战2：超长事务

**场景**：全局事务涉及大量数据，执行时间过长
```
问题1: Undo Log 存储量大
问题2: 其他事务等待行锁超时
问题3: TC ���连接池耗尽
```

**解决方案**：
- 分批处理（批量插入）
- 设置单语句超时
- 限制单事务影响行数

### 挑战3：服务不可用

**场景**：分支事务执行完成，但 TM/RM 与 TC 通信失败
```
问题: 无法判断事务最终状态
- TM 侧：收到超时，但可能已提交
- RM 侧：收到指令，但可能已处理
```

**解决方案**：
- TC 侧：定期扫描未提交事务，发起回滚
- RM 侧：定时检查 TC 状态，决定提交/回滚

### 挑战4：非事务操作

**场景**：调用第三方服务（非 Seata 管理）
```
问题: 无法回滚第三方操作
例: 调用微信支付API成功后全局事务失败
```

**解决方案**：
- 使用 **TCC 模式**（预留资源）
- 手动补偿（编写回调逻辑）
- 消息队列（最终一致性）

## ⑥ Select - 选型（选哪个）

### 候选方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| AT 模式 | 自动补偿，代码无侵入 | 需要Undo Log表 | 大多数场景 |
| TCC 模式 | 性能高，可控性强 | 需改代码 | 高并发场景 |
| SAGA 模式 | 长流程，编排灵活 | 无自动回滚 | 流程审批 |
| XA 模式 | 标准协议 | 长期锁 | 强一致性 |

### AT vs TCC 对比

| 维度 | AT | TCC |
|------|-----|-----|
| 代码侵入 | 无 | 需改三阶段代码 |
| 性能 | 一般 | 高 |
| 回滚 | 自动 | 手动 |
| 第三方支持 | 差 | 好 |
| 存储 undo log | 是 | 否 |

### 选型理由

**选 AT 模式，因为**：
1. **代码无侵入**：无需修改业务代码
2. **自动补偿**：自动生成回滚 SQL
3. **社区成熟**：Seata 文档全，社区活跃
4. **集成简单**：只需配置 DataSource 代理

### 核心代码

#### 1. 订单服务

```java
@GlobalTransactional(name = "create-order", timeout = 30000)
public void createOrder(OrderCreateReq req) {
    // 1. 创建订单
    Order order = new Order();
    order.setOrderNo(generateOrderNo());
    order.setUserId(req.getUserId());
    order.setAmount(req.getAmount());
    order.setStatus(0);
    orderMapper.insert(order);
    
    // 2. ��减库存
    stockService.deductStock(req.getItems());
    
    // 3. 扣减积分
    if (req.getPoints() > 0) {
        pointsService.deductPoints(req.getUserId(), req.getPoints());
    }
}
```

#### 2. 库存服务

```java
public void deductStock(List<OrderItem> items) {
    for (OrderItem item : items) {
        // AT 自动插入 undo_log
        stockMapper.deduct(item.getSkuCode(), item.getQuantity());
    }
}
```

### 失败恢复机制

```
场景1: 业务异常
  → 触发：抛出异常
  → 恢复：AT 自动回滚

场景2: TC 宕机
  → 触发：无法通信
  → 恢复：本地事务可正常提交

场景3: 网络闪断
  → 触发：重连
  → 恢复：TM 自动重试

场景4: 数据不一致
  → 触发：人工发现
  → 恢复：手动执行回滚脚本
```

### 约束条件

```
- 数据库：支持 InnoDB
- 事务长度：< 3 分钟
- 单次影响行数：< 10000
- 禁止：跨存储事务（Redis/MongoDB）
```

