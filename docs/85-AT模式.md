# 85-AT模式

> 分布式事务 - AT 模式核心原理

---

## ① Why - 价值

**背景与痛点**：
- 微服务架构下，多个服务间调用如何保证数据一致性？
- 传统分布式事务方案性能差、实现复杂
- 跨服务调用时，一个服务成功另一个失败，导致数据不一致

**收益**：
- 无侵入：业务代码无需改动，框架自动处理
- 高性能：一阶段提交本地事务后立即释放资源
- 自动回滚：通过回滚日志自动反向补偿

**用户**：
- 后端开发者、微服务架构师、DBA

---

## ② What - 定义

**一句话定义**：
Seata AT 模式是一种无侵入的分布式事务解决方案，基于两阶段提交协议演变而来，通过Undo Log实现自动回滚。

**核心组成**：
1. **TC (Transaction Coordinator)** - 事务协调器，维护全局事务状态
2. **TM (Transaction Manager)** - 事务管理器，开启/提交/回滚全局事务
3. **RM (Resource Manager)** - 资源管理器，管理分支事务与本地资源

**关键术语**：
- XID - 全局事务唯一ID，在服务间传递
- Undo Log - 回滚日志，记录前后镜像数据
- Global Lock - 全局锁，保证事务隔离性

## ③ How - 思维

**目标**：理解 AT 模式如何实现分布式事务

**范围**：
- yudao-cloud 项目中分布式事务的使用场景
- 支付模块、订单模块等需要事务一致性的场景

**禁止**：
- 禁止在生产环境随意开启全局事务
- 禁止大事务（影响性能）

**数据模型 - Undo Log 表**：

```sql
CREATE TABLE `undo_log` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT,
  `branch_id` bigint(20) NOT NULL,
  `xid` varchar(100) NOT NULL,
  `context` varchar(128) NOT NULL,
  `rollback_info` longblob NOT NULL,
  `log_status` int(11) NOT NULL,
  `log_created` datetime NOT NULL,
  `log_modified` datetime NOT NULL,
  PRIMARY KEY (`id`),
  UNIQUE KEY `ux_undo_log` (`xid`, `branch_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8;
```

**核心流程**：

```
全局事务开启(TM)
    ↓
分支事务执行(RM)
    ├─ 解析SQL
    ├─ 查询前镜像
    ├─ 执行业务SQL
    ├─ 查询后镜像
    ├─ 插入Undo Log
    ├─ 申请全局锁
    └─ 本地事务提交
    ↓
全局事务提交/回滚(TC)
    ├─ 提交：异步删除Undo Log
    └─ 回滚：读取Undo Log反向补偿
```

## ④ Hard - 难点

**难点1：全局锁死锁**
```
场景：tx1 持有 A 记录全局锁等待 B，tx2 持有 B 记录全局锁等待 A
→ 超时放弃，释放本地锁，让对方先执行
```

**难点2：脏读**
```
默认隔离级别：读未提交
→ 解决：SELECT FOR UPDATE 申请全局锁，保证读到已提交数据
```

**难点3：回滚失败**
```
场景：分支回滚时，另一事务仍在等待全局锁
→ 一直重试，直到对方超时放弃全局锁
```

**难点4：性能问题**
```
一阶段要申请全局锁，大量事务竞争
→ 解决方案：按业务拆分，降低竞争
```

## ⑥ Select - 选型

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| Seata AT | 无侵入、自动回滚、性能好 | 需要Undo Log表 | 大多数场景 |
| Seata TCC | 性能高、无Undo Log | 侵入性强需改代码 | 高性能要求 |
| Seata Saga | 长流程、可异步 | 无隔离性 | 流程审批 |
| 2PC/XA | 标准协议 | 性能差、需数据库支持 | 传统分布式 |

**选型理由**：
1. 无侵入：@GlobalTransactional 注解即可使用
2. 自动回滚：框架自动生成反向SQL
3. 社区活跃：Apache项目，文档完善
4. 与Spring Boot集成简单

**官方文档**：https://seata.io/zh-cn/

## ⑦ Impl - 实现

**使用方式 - 注解方式**：

```java
// 在需要分布式事务的方法上添加注解
@GlobalTransactional(rollbackFor = Exception.class)
public void createOrder(OrderDTO order) {
    // 1. 创建订单
    orderService.create(order);
    // 2. 扣减库存（远程调用）
    productService.deductStock(order.getItems());
    // 3. 扣减余额（远程调用）
    accountService.deductBalance(order.getUserId(), order.getAmount());
}
```

**yudao-cloud 中的实际使用**：

```java
// 支付模块 - 支付回调处理
@GlobalTransactional
public void handlePayNotify(PayNotifyDTO notifyDTO) {
    // 1. 更新支付订单状态
    payOrderService.updateStatus(notifyDTO.getOrderId(), PayStatusEnum.SUCCESS);
    // 2. 更新业务订单状态
    orderService.updateStatus(notifyDTO.getOrderId(), OrderStatusEnum.PAID);
    // 3. 通知用户
    notifyService.notifyUser(notifyDTO.getUserId());
}
```

**关键配置**：

```yaml
# application.yml
seata:
  enabled: true
  application-id: yudao-pay
  tx-service-group: my_test_tx_group
  registry:
    type: nacos
    nacos:
      server-addr: 127.0.0.1:8848
      namespace: public
```

**关键步骤校验**：

```
Step 1: 检查全局事务
  → 校验点：xid 已注册
  → 验证方法：日志查看 xid

Step 2: 执行分支事务
  → 校验点：Undo Log 已插入
  → 验证方法：查询 undo_log 表

Step 3: 提交/回滚
  → 校验点：TC 返回成功
  → 验证方法：控制台查看状态
```

**失败恢复机制**：

```
场景1：服务异常停止
  → 触发：TC 定期检查
  → 恢复：根据 Undo Log 自动回滚

场景2：网络中断
  → 触发：心跳检测失败
  → 恢复：TM 重试提交/回滚

场景3：全局锁超时
  → 触发：等待超时
  → 恢复：释放本地锁，重试
```

## ⑧ SKILL - 提炼

**触发条件**：
```
场景1：跨服务调用需要事务一致性
场景2：调用多个数据库需要保证原子性
场景3：异步调用需要最终一致性
```

**执行流程**：

```
Step 1: 引入依赖
  - 添加 seata-spring-boot-starter
  - 配置 TC 服务地址

Step 2: 开启全局事务
  - 在入口方法添加 @GlobalTransactional
  - 设置 rollbackFor = Exception.class

Step 3: 配置 undo_log 表
  - 在每个业务数据库创建 undo_log 表

Step 4: 验证
  - 模拟异常，验证自动回滚
  - 监控 TC 控制台查看事务状态
```
