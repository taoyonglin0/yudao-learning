# 11-配置版本管理.md

> 配置管理 - 版本管理与灰度发布

---

## ① Why - 价值 (为什么)

**背景与痛点**

- **背景**：在分布式系统中，配置不仅需要统一管理，还需要支持版本历史回滚、灰度发布等能力
- **不用它的痛点**：
  - 配置修改后无法回滚，一旦出错只能手动改回
  - 新配置无法先在小范围验证，直接全量发布风险高
  - 配置变更无审计，无法追溯谁在什么时候改了什么
  - 多环境（dev/test/pre/prod）配置切换麻烦，容易发错环境
- **收益**：
  - 配置版本历史可追溯，支持一键回滚
  - 灰度发布先验证，降低全量发布风险
  - 配置变更有审计日志
  - 多环境配置隔离与快速切换

**目标用户**

- 运维人员：配置回滚、灰度发布
- 开发人员：配置变更、版本预览
- 管理员：配置审计、变更审批

---

## ② What - 定义 (是什么)

**一句话定义**

配置版本管理与灰度发布是指对配置进行版本化存储、支持历史回滚，以及通过流量分配策略实现新配置在小范围内先验证再全量发布的能力。

**核心概念**

| 概念 | 说明 |
|------|------|
| 配置版本 | 配置每次变更生成一个新版本，保留历史记录 |
| 版本回滚 | 将配置恢复到之前的某个历史版本 |
| 灰度发布 | 先让少量用户/实例使用新配置，验证后再全量 |
| 流量匹配 | 根据请求特征（IP、Header、User-Agent等）决定走哪个版本 |
| 环境隔离 | dev/test/pre/prod 环境配置相互隔离 |

**核心流程**

```
配置变更 → 生成新版本 → 灰度发布(小流量验证) → 全量发布 → 异常回滚
```

## ③ How - 思维 (怎么做)

### 数据模型设计

```
配置版本表（infra_config_version）：
- id              : 主键
- config_key      : 配置键（如 database.url）
- version         : 版本号（自增）
- content         : 配置内容（JSON/YAML）
- status          : 状态（0灰度/1全量/2已回滚）
- release_ratio   : 灰度发布比例（0-100%）
- created_by      : 创建人
- created_time    : 创建时间
- released_time   : 发布时间
- remark          : 变更说明

灰度规则表（infra_gray_release）：
- id              : 主键
- config_key      : 配置键
- version         : 适用版本
- match_type      : 匹配类型（header/ip/userAgent）
- match_pattern   : 匹配规则（如 version=v2）
- created_time    : 创建时间
```

### 架构设计

```
┌──────────────┐    ┌──────────────┐
│  配置管理    │───▶│  版本管理    │
│  后台界面    │    │  服务       │
└──────────────┘    └──────────────┘
       │                   │
       ▼                   ▼
┌──────────────┐    ┌──────────────┐
│  数据库存储   │    │  灰度路由    │
│  (配置+版本)  │    │  (Gateway)   │
└──────────────┘    └──────────────┘
```

### Yudao 实现方式

**注意**：芋道项目采用简化实现：
- 配置存储在 `infra_config` 表（数据库），而非Nacos
- **灰度发布通过 Gateway 的 GrayLoadBalancer 实现**，基于请求Header中的version进行路由
- 配置版本管理通过数据库记录实现，每次更新生成新记录而非覆盖

```java
// 芋道配置实体 - 每次更新是新增，不是覆盖
@TableName("infra_config")
public class ConfigDO {
    private Long id;
    private String name;        // 配置名称
    private String configKey;    // 配置键
    private String configValue;  // 配置值
    private String type;         // 配置类型
    private Integer visible;     // 是否可见
    private String remark;      // 备注
    private Long creator;       // 创建人
    private Date createTime;    // 创建时间
    private Long updater;       // 更新人
    private Date updateTime;    // 更新时间
    private Boolean deleted;    // 是否删除
}
```

## ④ Hard - 难点 (挑战)

### 可能遇到的问题

**难点1：配置版本历史如何存储？**
```
场景：配置频繁变更，每次都存完整内容太占空间
→ 解决方案：只存储变更版本，差量存储；或定期清理历史版本
```

**难点2：灰度发布时流量如何精确控制？**
```
场景：想精确控制10%的流量，但网关随机路由可能不均匀
→ 解决方案：使用一致性Hash或权重路由，而非纯随机
```

**难点3：灰度验证失败如何快速回滚？**
```
场景：新配置有bug，灰度用户已经受影响
→ 解决方案：配置回滚秒级生效，支持一键切回上一版本
```

**难点4：多环境配置如何同步？**
```
场景：dev环境验证通过，想同步到test环境
→ 解决方案：支持配置导出/导入，或流水线自动同步
```

### 边界情况

- 灰度比例设置为0%时的处理
- 多个灰度规则同时匹配时的优先级
- 配置回滚时正在进行中的请求如何处理

## ⑥ Select - 选型 (选哪个)

### 候选方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| Nacos配置版本 | 原生支持版本历史、回滚、灰度 | 需要额外部署Nacos | 使用Nacos的项目 |
| Apollo配置中心 | 完整版本管理、灰度发布、审计 | 部署复杂，学习成本高 | 大型企业级项目 |
| Spring Cloud Config + Bus | 与Spring生态深度集成 | 版本管理较弱 | 传统Spring项目 |
| 数据库存储 + Gateway灰度 | 简单易实现，无需额外组件 | 版本管理需自己实现 | 芋道类项目 |

### 选型理由

**芋道选择方案：数据库存储 + Gateway灰度**

1. 无需额外部署配置中心，降低运维成本
2. 配置存储在数据库，与业务数据一起管理
3. 灰度发布通过Gateway实现，流量控制灵活
4. 与项目技术栈深度整合（Spring Boot + Gateway）

### 核心代码

**1. 配置服务（获取配置）**

```java
// ConfigServiceImpl.java
@Service
public class ConfigServiceImpl implements ConfigService {

    @Override
    public String getConfigValue(String configKey) {
        // 从数据库获取配置
        ConfigDO config = configMapper.selectByKey(configKey);
        return config != null ? config.getConfigValue() : null;
    }
}
```

**2. Gateway灰度路由（基于version header）**

```java
// GrayLoadBalancer.java（关键逻辑）
public class GrayLoadBalancer implements ReactorServiceInstanceLoadBalancer {

    private static final String VERSION = "version";

    @Override
    public Mono<Response<ServiceInstance>> choose(Request request) {
        // 从请求头获取version
        HttpHeaders headers = ((RequestDataContext) request.getContext())
            .getClientRequest().getHeaders();
        String version = headers.getFirst(VERSION);
        
        // 筛选匹配version的服务实例
        List<ServiceInstance> chooseInstances;
        if (StrUtil.isEmpty(version)) {
            chooseInstances = instances;  // 无version，走所有实例
        } else {
            // 只选择metadata中version匹配的实例
            chooseInstances = instances.stream()
                .filter(i -> version.equals(i.getMetadata().get("version")))
                .collect(Collectors.toList());
        }
        
        // 随机+权重选择实例
        return new DefaultResponse(NacosBalancer.getHostByRandomWeight3(chooseInstances));
    }
}
```

**3. 灰度发布配置示例**

```yaml
# application.yml - 服务实例打标
spring:
  cloud:
    nacos:
      discovery:
        metadata:
          version: v2    # 当前实例版本
```

```yaml
# Gateway路由配置
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service
          predicates:
            - Header=version, v2   # 只路由到v2版本
```

### 关键步骤校验

```
Step 1: 配置更新
  → 校验点：新配置写入数据库
  → 验证方法：查询数据库新记录

Step 2: 配置发布
  → 校验点：灰度实例标记version标签
  → 验证方法：检查Nacos/注册中心metadata

Step 3: 流量路由
  → 校验点：带version header的请求路由到对应实例
  → 验证方法：日志统计流量分布

Step 4: 回滚操作
  → 校验点：切换version标签或移除灰度实例
  → 验证方法：请求验证流量切换
```

### 约束条件

- 不引入新的外部依赖（Nacos已有）
- 保持API兼容，不改变接口签名
- 灰度生效时间<1秒
- 支持最多10个版本历史存储

## ⑧ SKILL - 提炼 (复用)

### 触发条件

```
场景1：需要配置版本历史和回滚功能
场景2：需要进行灰度发布验证新配置
场景3：需要按流量比例控制发布范围
```

### 执行流程

```
Step 1: 配置存储
  - 使用数据库存储配置（infra_config表）
  - 每次更新创建新记录而非覆盖

Step 2: 灰度路由配置
  - Gateway集成GrayLoadBalancer
  - 服务实例添加version metadata标签

Step 3: 流量控制
  - 请求带version header进行灰度路由
  - 无version header则走默认实例

Step 4: 回滚操作
  - 修改实例version标签或下线灰度实例
  - 流量自动切换到稳定版本
```
