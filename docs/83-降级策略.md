# 83-降级策略

> 服务保护 - 降级策略核心原理与实现

---

## ① Why - 价值

**背景与痛点**：
- 下游服务挂了，上游服务还在傻傻等待，最终拖垮自己
- 非核心服务故障导致核心业务流程中断
- 依赖多个服务，一个超时全部失败

**收益**：
- 快速失败，保护主流程
- 返回降级数据，保证核心功能可用
- 防止故障扩散，避免雪崩

**用户**：
- 微服务调用方、非核心业务流程

---

## ② What - 定义

**一句话定义**：
降级策略是当服务不可用时，提供备选方案的容错机制，保证核心功能可用。

**核心组成**：
1. **降级条件** - 超时、异常、熔断
2. **降级方案** - 返回默认值、缓存数据、友好提示
3. **恢复策略** - 自动恢复、人工介入

**关键术语**：
- Fallback - 降级处理方法
- FeignClient - 服务调用接口
- Circuit Breaker - 熔断器

---

## ③ How - 思维

**目标**：实现服务降级能力

**范围**：
- 允许：yudao-module-*/api/、yudao-framework/
- 禁止：改动业务核心逻辑

**降级模型**：

```java
// FeignClient 降级配置
@FeignClient(name = "service-name", fallbackFactory = XxxFallbackFactory.class)
public interface XxxApi {
    // 服务方法
}

// FallbackFactory 实现
@Component
public class XxxFallbackFactory implements FallbackFactory<XxxApi> {
    @Override
    public XxxApi create(Throwable cause) {
        return new XxxApi() {
            @Override
            public CommonResult<String> getData() {
                // 降级返回
                return CommonResult.success("降级数据");
            }
        };
    }
}
```

**降级流程**：
```
服务调用 
  → 正常→返回结果 
  → 超时/异常→触发降级 
  → 返回降级数据
```

## ④ Hard - 难点

**难点1：降级数据从哪来？**
```
场景：服务挂了，返回什么数据？
→ 预定义默认值、读缓存、兜底接口
```

**难点2：降级时机如何判断？**
```
场景：什么时候该降级？
→ 超时、连续失败达到阈值、熔断打开
```

**难点3：如何防止降级被滥用？**
```
场景：所有接口都配降级，维护困难
→ 区分核心/非核心，只对非核心配置
```

## ⑥ Select - 选型

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| Feign + Hystrix | Spring原生 | 已停更 | 老项目 |
| Feign + Sentinel | 阿里出品功能全 | 较重 | 阿里技术栈 |
| Resilience4j | 轻量现代 | 需额外集成 | 新项目 |
| 手动降级 | 简单直接 | 不灵活 | 简单场景 |

**选型理由**：
1. yudao使用Feign，当前预留fallbackFactory接口
2. 可集成Sentinel实现更丰富降级策略
3. 降级策略需要根据业务定制

## ⑦ Impl - 实现

**当前实现**：

yudao-cloud 中 FeignClient 预留了降级接口：

```java
// 源码中的TODO：fallbackFactory
@FeignClient(name = ApiConstants.NAME) // TODO 芋艿：fallbackFactory =
public interface DictDataApi extends DictDataCommonApi {
    // 服务方法
}
```

**实现示例**：

```java
// 1. 定义降级工厂
@Component
public class DictDataFallbackFactory implements FallbackFactory<DictDataApi> {
    @Override
    public DictDataApi create(Throwable cause) {
        return new DictDataApi() {
            @Override
            public CommonResult<Boolean> validateDictDataList(String dictType, Collection<String> values) {
                // 降级：直接返回校验通过
                log.warn("[DictDataFallback][字典数据降级返回，dictType({})]", dictType);
                return CommonResult.success(true);
            }
        };
    }
}

// 2. 配置FeignClient
@FeignClient(name = "system-service", fallbackFactory = DictDataFallbackFactory.class)
public interface DictDataApi {
    // ...
}

// 3. 超时配置
feign:
  client:
    config:
      default:
        connectTimeout: 5000
        readTimeout: 5000
```

**降级方案分类**：

| 类型 | 说明 | 示例 |
|------|------|------|
| 返回默认值 | 返回预设常量 | `success(false)` |
| 返回缓存 | 读取最近缓存 | 缓存30秒内的数据 |
| 友好提示 | 返回可读消息 | "服务暂不可用" |
| 托底接口 | 调用备用服务 | 降级到简化接口 |

## ⑧ SKILL - 提炼

**触发条件**：
- 场景1：依赖的下游服务不可用
- 场景2：服务响应超时
- 场景3：非核心业务流程

**执行流程**：
```
Step 1: 定义FallbackFactory
  - 实现FallbackFactory接口
  - 创建降级方法

Step 2: 配置FeignClient
  - 添加fallbackFactory属性
  - 注册到Spring容器

Step 3: 配置超时
  - 设置connectTimeout
  - 设置readTimeout

Step 4: 验证降级
  - 模拟服务不可用
  - 验证降级数据返回
```

**配方**：
- 技术栈：Java 8+, Spring Cloud Feign
- 依赖：spring-cloud-starter-openfeign
- 注解：@FeignClient、@FallbackFactory

**验收标准**：
- [ ] 降级方法正确：服务不可用时返回降级数据
- [ ] 性能达标：降级响应<50ms
- [ ] 日志可查：降级触发有日志记录
- [ ] 可关闭：可通过配置禁用降级
